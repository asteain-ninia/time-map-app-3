## 6. 階層モデルの新仕様 (2026-04-28 確定)

### 6.1 設計の経緯

旧 §6 は「グローバル `Layer` 概念を残し、`Layer.order` 序列とコンテナ rings 空化で階層整合性を担保する」方針だったが、2026-04-28 の設計議論で以下の構造的問題が露見し全面廃棄された:

- レイヤー深さが国ごとに異なるとき (例: 3層国 A vs 4層国 B)、グローバル序列ではリーフ排他を正しく判定できない (異なるツリーのリーフ同士が「下に潜り込める」)
- 「全リーフが何らかのコンテナに包まれる」と要求しても上記問題は解決しない

結論として、**グローバルレイヤー概念そのものを廃止し、ツリー位置から派生する depth で階層を表現する**方針へ転換する。設計意図は「基礎自治体 (リーフ) を面地理情報の唯一の担い手とし、上位はその集合体である純粋なコンテナ。階層深さは可変 (1層〜何十層)。歴史上の領土変更類型を全部カバーする」。

### 6.2 データモデル

- **`Layer` エンティティを廃止する**
- **`FeatureAnchor.placement.layerId` を削除する**
- **`FeatureAnchor.placement` に `isTopLevel: boolean` (最上位フラグ) を追加する** (時間依存。所属変更フローで時刻ごとに変化可能。同一錨内で `isTopLevel === (parentId === null)` 不変条件)
- **`FeatureAnchor.shape` を optional にする** (リーフのみ持つ。コンテナは shape プロパティ自体なし)
- **`FeatureAnchor.property` に `kind` プロパティを追加する** (国 / 州 / 郡 / 町 / 連邦 / 植民地 などの種別ラベル。表示専用で整合性検証には関与しない。錨ごとに保持することで時間軸上の階級変動 — 例: 同一地物の「植民地 → 州」「国 → 帝国 → 共和国」 — を表現可能)
- **既存 `.gimoza` 互換性は破棄する** (移行マイグレーションは実装しない、ユーザー判断済み)

### 6.3 派生概念: depth

- depth は **対象時刻における当該地物の有効錨のツリー位置から派生**: ルート (有効錨の `placement.isTopLevel === true && placement.parentId === null`) は `depth = 0`、子は親地物の同時刻有効錨から導出した depth + 1
- depth は地物自身に保持しない (常に算出する派生値)
- 不変条件: 同一錨内で `placement.isTopLevel === (placement.parentId === null)`
- **用語の固定**: `depth = 0` が **最上位 (国側、上)**。depth 大が下位 (基礎自治体側、下)。ユーザー感覚として「上 = 国 / 下 = 町」
- データ構造は時刻ごとに評価されるツリーの森 (複数ルート)。最上位フラグは錨側に持つため、所属変更フローで時刻ごとに位相変化（独立・帰属・連邦化など）を表現できる

### 6.4 整合性ルール

- **リーフ排他は地図全体で適用** (同一親の子同士に限定せず、異なるツリーのリーフ同士も含む)
- **親 ≡ 子の和 (厳密一致)**。コンテナ内部の隙間は許容しない
    - 直轄領のような隙間ケースは明示的なリーフを置いて埋める (例: 「政府直轄-県」)
    - **リーフ → コンテナ遷移時の差分自動処理**: 所属変更フローでリーフが子を獲得しコンテナ化する瞬間、元リーフの `shape` と新規子の和に差分があれば、差分を自動的に「直轄領リーフ」として生成し新コンテナの直接子に追加する。デフォルト名は「`<元リーフ名> 直轄領`」、`kind` は元リーフの遷移時刻における該当錨の `property.kind` を継承（直轄領の最初の錨に設定）。差分が複数の領土に分かれる場合は1リーフに複数 `territory` リング (飛び地化ロジック) でまとめる。差分が空なら直轄領を生成しない。所属変更ダイアログでユーザーが直轄領の名前/種別を上書き可能
    - 既存コンテナへの子追加 (所属変更フローによる追加のみ) では、コンテナ自身が `shape` を持たないため差分処理は発生しない (形状は子の和として再計算されるだけ)。混在結合では結果コンテナが新規生成されるため、既存コンテナへの追加には該当しない
- 地図全体としての無主地 (どのリーフにも覆われない領域) は許容する
- リーフ判定は「`shape` を保持し、子を持たない」を必要十分条件とする
- コンテナの形状 (描画 / ヒットテスト / 隣接判定) はすべて子の和集合として実行時に導出する
- **下位領域獲得経路**: 既存地物が子を獲得するのは「所属変更フロー」のみ。分裂・結合 (リーフのみ・コンテナのみ・混在のいずれのパターン) は新地物の生成または既存地物の形状変化を扱い、既存地物に後から子を加える経路ではない。混在結合では結果コンテナが新規生成され、混在リーフはその新規コンテナの子として帰属する (元コンテナの同一性は終了)。これは**結果地物（統合地物）の生成を結合の3パターン全てで「新規生成」に統一する**設計判断 (Undo/Redo・履歴の錨・ID継続の実装方針を結合機能で一致させる)。なお混在パターンでは選択された末端地物は存在継続し新統合集約地物の下位領域へ取り込まれるため、選択地物の総数は減らない場合がある（減るのは「結合の主体」となる地物の数）
- **結合対象の選択集合制約**: 結合対象は互いに上位・下位関係にあってはならない (集約地物 A と、A の直接・間接いずれかの下位領域に含まれる地物 X を同時選択することは禁止)。下位関係にある地物の同時選択は重複 child 参照や不正なツリー更新の原因となるため、結合実行時に検出し操作を拒否する

### 6.5 操作モデル

- 「**征服 (既存親への割譲)**」と「**連邦化 (新規共通親の作成)**」は **同じ所属変更フローで表現する**
- 新規親作成ループは現行 `ParentTransferDialog` の延長として実装する:
    - 名前 / 種別 (`kind`) 入力
    - 末端で「**既存最上位を選ぶ**」or「**新規最上位を作成する** (`isTopLevel: true`)」を選択
    - 再帰的に親を積み上げ可能 (中間階層を後付けで挿入できる)
- 親候補は depth 制約ではなく「対象地物自身および対象地物の上位領域すべて (直接の親、その親、…と最上位まで連鎖して辿った上位領域の集合) に含まれないこと (循環防止) + 親 ≡ 子の和 制約を将来満たせること」で絞る

#### 例: アメリカ合衆国爆誕シナリオ

1. 時刻 T において、13 個の最上位国 (各々の T 時点有効錨で `placement.isTopLevel: true` / `parentId: null`) を複数選択
2. 「親変更」(現行の所属変更フロー) を起動
3. 親候補として「**新規地物を作成**」を選択
4. 新規親作成フローで「合衆国」を新規地物として作成し、時刻 T で開始する錨に `placement.isTopLevel: true` / `parentId: null` を設定
5. 13 国の各々に時刻 T で開始する新しい錨を追加し、`placement.isTopLevel: false` / `parentId: 合衆国ID` を設定（時刻 T 未満の錨は `isTopLevel: true` / `parentId: null` のまま継続）。13 国の同一性（Feature ID）は維持されたまま、時間軸上で位相のみが変化する
6. 全ステップを 1 Undo にまとめる

### 6.6 自動配色

- **同 depth の兄弟で 4 色塗り分け** (同じ親を持つ子同士の中で 4 色定理を適用)
- 個別 feature が optional `style` を持つ。未指定なら自動配色を継承

### 6.7 表示フォーカス (戦略ゲー風)

- **グローバル depth スライダー**: 全地物を `depth ≤ N` で集約表示 (N = 0 で最上位ルートだけ、N 増で深く展開)
- **個別フォーカス** (グローバルと併存): 「選択地物の縦方向領域 (上位領域すべて + 自身 + 下位領域すべて) だけ depth M、その他は depth N まで」のように選択した地物の縦方向領域だけ深掘りする
- 想定 UI 参考: 戦略ゲー (HoI4 / EU4 等) の depth 集約表示
- 縦方向領域の絞り込み: 選択中地物の上位領域すべて + 自身 + 下位領域すべて (縦方向に繋がる領域) だけを集約対象として提示し、認知負荷を下げる
- 性能上、空間インデックス (R-tree など) の導入が必要になる可能性 (Phase 概略の後段で評価)

### 6.8 UI 整理 (`kind` ラベル)

- レイヤー名と地物名の混合 (例: 県レイヤーの「東京」が「東京都-県」になる) を避けるため、`kind` は **表示用ラベルとして錨側 (`FeatureAnchor.property.kind`) に持つ** (整合性検証には関与しない。時間軸上の階級変動を表現可能)
- 種別の制約: 親が「子の種別」を強制しない (例: 日本のように県の下に市町村が混ざる柔軟性を維持)
- Sidebar の現行 `レイヤー / プロパティ / 地物一覧` 3タブ構成は廃止または再構成する。具体的なタブ構成は実装着手時にユーザー判断:
    - 「選択地物中心」のタブ (プロパティ + 縦方向領域ナビゲーション = 上位領域・自身・下位領域の連鎖を辿る UI)
    - 「世界俯瞰」のタブ (地物一覧 + 深度フォーカス制御)
    - 種別管理 (`kind` の命名規則) はプロジェクト設定または独立タブへ

### 6.9 想定する歴史的領土変更類型

新モデルで以下の類型がすべて表現できることを確認する。個別 UI を設けるのではなく、データモデルとして表現可能であれば良い。

#### A. ツリー位相変化

| 類型 | 操作 |
| --- | --- |
| 改名・属性変更 | プロパティ編集 |
| 国境調整 | リーフの頂点編集 |
| 割譲 | 所属変更 (新親 = 既存地物) |
| 独立 | 所属変更 (`isTopLevel = true` へ昇格) |
| 帰属 | 所属変更 (新親 = 既存地物) |
| 直轄化 (中間階層スキップ) | 所属変更で中間コンテナを抜かして、より上位の上位領域 (直接の親の親、その親、…) へ移す |
| 自治化 (中間階層挿入) | 新規親作成 (既存子と旧親の間に新コンテナを挿入) |

#### B. 数量変化

| 類型 | 操作 |
| --- | --- |
| 分裂 (二分割) | ナイフ分割。**元リーフは形状変化として継続 (プロパティ引き継ぎ先) + もう片方が新規リーフとして生成** (穴あけと同型)。元地物終了+新2地物の旧仕様からの変更点に注意 (現行実装は旧仕様。Phase 4 で移行) |
| 結合 (リーフのみ) | 結合。形状の論理和。元リーフ全消滅+結果リーフ新規生成 |
| 結合 (コンテナのみ平坦化) | 結合。元コンテナそれぞれの直接子を新コンテナの直接子へ平坦化統合 (中間階層は保持しない)。階層保持は所属変更フローで表現 |
| 結合 (リーフ+コンテナ混在の自動帰属) | 結合。混在リーフを新コンテナの直接子として帰属。リーフは存在継続、コンテナは新規生成 (旧コンテナは終了) |
| 連邦化 (複数独立国 → 共通親配下に統合) | 所属変更フローで「親候補 = 新規地物作成」を選択 (§6.5)。**結合ではなく所属変更フローを使う**。結合は結合の主体（リーフのみは選択リーフ全体、集約のみは選択集約全体、混在は選択集約のみ）の同一性を終了させ結果地物を新規生成する操作、所属変更は地物数を保ったまま位相を変える操作という設計分離 |
| 解体 | 全子を `isTopLevel = true` へ昇格。コンテナは子全消失で自動消滅 |

#### C. リーフ ↔ コンテナ遷移

| 類型 | 操作 |
| --- | --- |
| リーフ拡大 (X が拡大しもとの X が一単位として残る) | 新規親作成 (新コンテナ Y を作って既存リーフ X をその子に) |
| コンテナ自動消滅 | 子全消失で自動 (既存挙動) |

#### D. 階層変動 (深さ変化)

| 類型 | 操作 |
| --- | --- |
| 階級変動 | `kind` プロパティ更新 (depth は親変更で従う) |
| 多層国 ↔ 1層国の編入・編出 (深さ非対称) | 所属変更 + 必要なら `kind` 変更 |
| 構造改革 (道州制等の大規模再編) | 複数の新規親作成 + 所属変更の連続適用 |

#### E. 政治的特殊形態 (単一親モデルで近似)

「高々 1 つの上位コンテナに属する」制約を維持したまま、別エンティティ化や時間付き所属切替で表現する。

| 類型 | 表現方法 |
| --- | --- |
| 連邦 (例: 米合衆国、EU、神聖ローマ帝国) | 連邦エンティティをコンテナとして最上位に置き、構成国・州を子に。`kind = "連邦"` などで政体名を表現 |
| 二重君主国 (例: オーストリア = ハンガリー) | 連邦と同じく上位コンテナで表現 (オーストリア = ハンガリーがコンテナ、オーストリア・ハンガリーが子) |
| 同君連合 (例: ハプスブルク朝) | 領土変更ではなく統治者メタデータ。プロパティで表現 |
| 共同統治 (例: 英埃領スーダン、アンドラ) | 「○○共同統治領」として独立した最上位エンティティ。宗主はプロパティで列記 |
| 占領地 (例: 第二次大戦中のヴィシー政権下フランス) | 「○○国占領地」として戦時独立エンティティ + 基礎自治体は時間ごとに所属親を切替 (時間付き `FeatureAnchor.placement.parentId` で対応) |
| 係争地 (例: 国家承認が分かれる地域) | 「○○ (係争中)」として独立した最上位エンティティ + 主張する側はプロパティで列記 |

### 6.12 用語と内部実装識別子の対応

要件定義書 (§2.1 改訂時) は仕様本文で和訳語のみを使用し、英語識別子は内部実装補遺へ集約する (開発ガイド §重要な注意事項 の規約)。新 §6 で導入する概念の対応表を以下に固定する。既存仕様で確立済みの語 (歴史の錨 / 親領域 / 下位領域 / リング / 領土リング / 穴リング / 飛び地 / リーフノード) はそのまま継承する。

#### 概念・データモデル

| 仕様用語 (和訳) | 内部実装識別子 | 補足 |
| --- | --- | --- |
| 地物 | `Feature` | 既存訳の継承 |
| 歴史の錨 | `FeatureAnchor` | 既存訳の継承 |
| 形状 | `FeatureAnchor.shape`? (要件定義書 §4.1 で optional 化確定。Phase 2 でコード反映予定) | 新モデルでは optional (末端地物のみ保持)。現行コードは required。Phase 2 で型変更 |
| 親領域 (親) | `parent` / `FeatureAnchor.placement.parentId` | 既存訳の継承 |
| 下位領域 (子) | `childIds` / `FeatureAnchor.placement.childIds` | 既存訳の継承 |
| 最上位フラグ | `FeatureAnchor.placement.isTopLevel` (要件定義書 §4.1 で識別子確定。Phase 2 でコード実装予定) | 真のとき当該錨の有効期間において親領域を持たないルート地物。時間依存（同一地物でも錨ごとに異なる）。実コード未実装 |
| 種別 | `FeatureAnchor.property.kind`? (要件定義書 §4.1 で識別子確定。Phase 2 でコード実装予定) | 表示専用ラベル (国 / 州 / 郡 / 町 / 連邦 / 植民地 など)。整合性検証に関与しない。錨側に保持（時間依存）。実コード未実装 |
| 末端地物 (リーフ) | leaf (概念名) | 形状を保持し下位領域を持たない地物。「リーフノード」既存訳の継承。コード上の専用型は未定 (Phase 3 で `Feature` 型ガードまたは派生型として実装予定) |
