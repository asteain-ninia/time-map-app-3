# ドキュメントレビューFB

作成日: 2026-03-25
対象リポジトリ: `time-map-app-3`
対象文書:

- `指示.md`
- `技術方針.md`
- `開発ガイド.md`
- `現状.md`
- `要件定義書.md`

---

## 1. 結論

このリポジトリのドキュメント群は、量は十分あります。しかし、現時点では「多い」ことがそのまま「使える」ことに繋がっていません。

最も大きい問題は次の3点です。

1. **文書ごとの役割分担が崩れている**
2. **文書同士のルールが衝突している**
3. **実装と一致していない記述が、完了・未実装の判断文書に混入している**

特に `現状.md` は、現状把握のための文書としての信頼性がかなり落ちています。  
理由は、未実装一覧・完了履歴・進捗記録・既知不具合・ロードマップが一つの文書に混ざり、しかも相互に矛盾しているからです。

この状態では、実装者が文書を読んでも、

- 何が本当に未実装なのか
- 何が仕様で何が設計方針なのか
- 何が「今の事実」で何が「将来構想」なのか

を即座に判断できません。

短く言えば、今の課題は「文書が足りない」ことではなく、**文書体系が情報アーキテクチャとして破綻し始めている**ことです。

---

## 2. このレビューの観点

今回は、文書を以下の観点でレビューしました。

### 2.1 正確性

- 実装と一致しているか
- 完了/未実装の記述が事実と合っているか

### 2.2 一貫性

- 文書同士でルールが衝突していないか
- 同じ対象について別の文書で逆のことを書いていないか

### 2.3 運用性

- 実装者が読んで次の行動を判断できるか
- 更新ルールが実際に守れる設計になっているか

### 2.4 境界の明確さ

- 仕様・設計・実装方針・進捗記録・障害記録が適切に分離されているか

## 4. 最重要の指摘

## 4.1 `現状.md` が「現状の真実源」になれていない

これは最重要です。

`開発ガイド.md` では、`現状.md` について次のように定義しています。

```md
- 現状.mdは「未解決の課題」だけを管理すること。完了した不具合は原則として即時削除すること。
```

一方で、`指示.md` では次のように書かれています。

```md
- **テスト通過後、現状.mdの更新を忘れないこと。** 実装コミットごとに、実装済みテーブル・未実装テーブル・ロードマップ・テスト総数を更新する。まとめて後から更新しない。
```

この2つは、同時には運用できません。

### 何が矛盾しているか

`開発ガイド.md` の思想は、

- `現状.md` = 未解決課題専用

です。

しかし `指示.md` の思想は、

- `現状.md` = 実装済み一覧 + 未実装一覧 + ロードマップ + テスト統計

です。

つまり、`現状.md` に対して **課題管理台帳** と **プロジェクト全体台帳** の2つの役割を同時に要求しています。

### 実際に何が起きているか

現在の `現状.md` はその両方を抱え込んでいるため、以下のような自己矛盾が発生しています。

#### 例1: レイヤー削除

未実装一覧:

```md
| レイヤーの削除（地物未所属時のみ） | 中 | 要件定義書§2.1に仕様追記済み。ManageLayersUseCaseに削除メソッド未実装 |
```

完了履歴:

```md
28. ~~**レイヤー削除** — ManageLayersUseCaseに削除メソッド追加。地物未所属時のみ（§2.1）~~ ✅ 完了（4テスト）
```

同じ機能が「未実装」と「完了」に同時に存在しています。

#### 例2: 競合解決ダイアログ

未実装一覧:

```md
| 競合解決ダイアログ | 低 | ミニマップ付き。UI未実装 |
```

実装済み一覧:

```md
| `ConflictResolutionDialog.svelte` | presentation/components/ | ✅ 完了 | 競合解決ダイアログ。競合リスト・優先地物選択・ナビゲーション・解決適用 |
```

完了履歴:

```md
45. ~~**競合解決ダイアログUI**（§2.2.3）~~ ✅ 完了 ...
```

これも同じです。

#### 例3: プロパティタブ自動切替

E2E fixme一覧:

```md
| `e2e/edit-mode.spec.ts` | 7(1fixme) | §2.3.3/§2.4.3: 地物選択→プロパティタブ(fixme) ...
```

完了履歴:

```md
49. ~~**Sidebarプロパティタブ** + 属性編集フォーム ... 地物選択時にプロパティタブへ自動切替）
```

これは「文書中のコンフリクト」です。

### なぜ悪いか

この状態になると `現状.md` を読んでも、次のような判断ができません。

- 直すべき未実装は何か
- もう終わっているはずの項目はどれか
- E2E の fixme は既知不具合なのか、文書の更新漏れなのか

結果として、`現状.md` は現状把握文書ではなく、**情報の衝突点**になっています。

### 改善方針

`現状.md` は次のどちらかに役割を固定すべきです。

#### 案A: 未解決課題専用に戻す

- 未実装
- 既知不具合
- 残作業

のみを書く。

その場合、以下は別文書へ追い出すべきです。

- 実装済み一覧
- フェーズ履歴
- 完了履歴
- テスト統計サマリ

#### 案B: プロジェクト全体ステータス台帳として再定義する

この場合は `開発ガイド.md` 側を修正し、

- 未解決課題
- 実装済み
- 進捗履歴
- 既知不具合

を同居させることを正式化します。

ただしこの場合でも、少なくとも「同一項目が未実装と完了に二重記載されない構造」へ再編する必要があります。

## 4.2 `指示.md` と `開発ガイド.md` のルール衝突が、運用破綻の根本原因になっている

これは `現状.md` 問題の親です。

### `指示.md`

```md
- 既存コードがある論点では、根拠となるソースコードを直接引用してください。
...
- **テスト通過後、現状.mdの更新を忘れないこと。** 実装コミットごとに、実装済みテーブル・未実装テーブル・ロードマップ・テスト総数を更新する。
```

### `開発ガイド.md`

```md
- 現状.mdは「未解決の課題」だけを管理すること。完了した不具合は原則として即時削除すること。
```

### 問題

この衝突は単なる表現ブレではありません。

ここがズレていると、AI・人間どちらが更新しても必ず以下のどちらかになります。

- ガイド違反
- 指示違反

つまり、**更新すればするほどどちらかに反する**状態です。

### なぜ悪いか

文書運用ルールは、実装ルールよりも厳密でなければいけません。  
理由は、文書ルールが曖昧だと、以後の履歴すべてが汚染されるからです。

### 改善方針

最優先で、以下を決める必要があります。

1. `現状.md` は「課題管理」なのか「総合進捗台帳」なのか
2. 完了履歴を残すのか残さないのか
3. テスト総数を `現状.md` に置き続けるのか、別文書へ分離するのか

これを決めない限り、他の文書を直しても再び崩れます。

## 4.3 `現状.md` の「完了」判定が UI 存在ベースで、機能完成と混同されている

これも重要です。

例として `ProjectSettingsDialog` の扱いを見ます。

`現状.md` では次のように完了扱いです。

```md
| `ProjectSettingsDialog.svelte` | presentation/components/ | ✅ 完了 | プロジェクト設定ダイアログ。プロジェクト名/説明、タイムライン範囲、惑星パラメータ、グリッド設定、面スタイル設定 |
```

また、完了履歴でもこう書かれています。

```md
52. ~~**プロジェクト設定ダイアログ**（§2.6）~~ ✅ 完了 ...
```

しかし実装を見ると、保存処理は `App.svelte` 内のローカル state 更新で止まっています。

`src/renderer/App.svelte`

```ts
function onSettingsSave(metaPatch: Partial<WorldMetadata>, settingsPatch: Partial<WorldSettings>): void {
  projectMetadata = { ...projectMetadata, ...metaPatch };
  projectSettings = { ...projectSettings, ...settingsPatch };
  projectMetadata = { ...projectMetadata, settings: projectSettings };
}
```

さらに保存時は `SaveLoadUseCase` が metadata を固定生成しています。

`src/renderer/application/SaveLoadUseCase.ts`

```ts
assembleWorld(): World {
  const metadata: WorldMetadata = {
    ...DEFAULT_METADATA,
    sliderMin: 0,
    sliderMax: 10000,
  };
```

読み込み時も metadata を戻していません。

```ts
private distributeWorld(world: World): void {
  this.addFeature.restore(world.features, world.vertices, world.sharedVertexGroups);
  this.manageLayers.restore(world.layers);
  // メタデータのスライダー範囲は将来のNavigateTimeUseCase拡張で対応
}
```

### 問題

つまり、`ProjectSettingsDialog` は

- ダイアログ表示はある
- 入力欄もある
- だが永続化も復元も実反映も閉じていない

という状態です。

この状態で「完了」と書くのは、厳密には誤りです。

### 同種の問題

同様に `現状.md` では以下も過大評価されています。

```md
56. ~~**未保存変更警告** + D&D読み込み + 自動バックアップ（§2.5）~~ ✅ 完了 ...
```

しかし実装ではバックアップで `window.fileAPI` を呼んでいます。

`src/renderer/App.svelte`

```ts
const content = await window.fileAPI.readFile(fromPath);
await window.fileAPI.writeFile(toPath, content);
await window.fileAPI.writeFile(getBackupFileName(filePath, 1), json);
```

一方 preload が公開しているのは `window.api` です。

`src/preload/index.ts`

```ts
contextBridge.exposeInMainWorld('api', api);
```

`src/preload/index.d.ts`

```ts
interface Window {
  api: FileAPI;
}
```

ここから分かるのは、`現状.md` の「完了」は必ずしも「仕様どおり動作確認済み」ではなく、しばしば「UIやコード片が存在する」という意味に落ちていることです。

### 改善方針

`現状.md` の状態ラベルは最低でも次の粒度に分けるべきです。

- 未着手
- 部分実装
- UIのみ
- ロジックのみ
- 接続済み
- 検証済み
- 完了

今の `✅ 完了` は意味が広すぎます。

## 4.4 `開発ガイド.md` の一部が古く、実装者を誤誘導する

### 問題例: `useMachine`

`開発ガイド.md` には次の記述があります。

```md
- Svelteコンポーネントは `useMachine` を通じてステートマシンを購読し、ユーザー操作はイベントとしてマシンに送信する
```

しかし現実の実装はそうなっていません。

`src/renderer/presentation/state/toolStore.ts`

```ts
 * @xstate/svelte は Svelte 3/4 向けのため手動統合を行う。
```

```ts
import { createActor } from 'xstate';
```

つまり今の正しい知識は、

- `useMachine` ではない
- `@xstate/svelte` ではない
- `createActor` ベースの手動統合

です。

### なぜ悪いか

これは単なる細部ではありません。  
新しい実装者やAIが `開発ガイド.md` を信じて実装を始めると、**現在のコードベースと異なる作法でコードを書き始める**危険があります。

### 改善方針

ここは以下のように書き換えるべきです。

- XState v5 は `createActor` ベースで手動統合する
- `@xstate/svelte` は現プロジェクトでは採用しない
- 理由: Svelte 5 / 既存実装との整合

## 4.5 `技術方針.md` が「現状説明」と「理想設計」を混同している

`技術方針.md` は本来、アーキテクチャの基準文書です。  
そのため、現状の実装を正確に説明するのか、到達目標を書くのかを明確にしなければいけません。

しかし今は両方が混ざっています。

### 例1: DIコンテナ

`技術方針.md`

```md
- 全サービスの初期化をDIコンテナで一元管理
```

しかし実ランタイムでは `appState.ts` がグローバルシングルトンを export しています。

`src/renderer/presentation/state/appState.ts`

```ts
export const manageLayers = new ManageLayersUseCase();
export const navigateTime = new NavigateTimeUseCase();
export const addFeature = new AddFeatureUseCase();
```

DIContainer 自体は存在します。

`src/renderer/infrastructure/DIContainer.ts`

```ts
export class DIContainer {
```

```ts
export function getContainer(): DIContainer {
```

つまり現状は、

- 技術方針: DI 一元管理
- 実装実態: グローバルシングルトン運用

です。

### 例2: ラベル描画責務

`技術方針.md`

```md
- LabelRenderer: ラベル配置
```

しかし実装は `FeatureRenderer.svelte` に内包されています。

`src/renderer/presentation/components/FeatureRenderer.svelte`

```ts
const LABEL_MIN_ZOOM = 2;
```

```svelte
{#if anchor.property.name && zoom >= LABEL_MIN_ZOOM}
  {@const centroid = getCentroid(anchor)}
  {#if centroid}
    <text ...>
      {anchor.property.name}
    </text>
  {/if}
{/if}
```

### 例3: 200行以下方針

`技術方針.md`

```md
- コンポーネント分割が自然で、各ファイルを200行以下に保ちやすい
```

しかし実ファイル規模は次のとおりです。

- `src/renderer/App.svelte`: 1146 行
- `src/renderer/presentation/components/MapCanvas.svelte`: 556 行
- `src/renderer/infrastructure/persistence/JSONSerializer.ts`: 548 行

### 問題

方針書が

- 現在の事実
- 守るべき原則
- 将来の目標

を区別せずに書いているため、読む側が「これは現状か、理想か」を判断できません。

### 改善方針

`技術方針.md` は各節ごとに以下を分けるべきです。

- **採用方針**
- **現状実装**
- **未達の課題**

これだけで読みやすさと正確性が大きく上がります。

## 4.6 `要件定義書.md` の「本文と実装補遺の分離方針」は概ね良いが、運用はまだ甘い

`要件定義書.md` 冒頭には、非常に良いルールがあります。

```md
- 仕様本文では、英語識別子やコード値の直接記載を原則禁止とする。
- 内部実装補遺でのみ、実装上必要な識別子・キー名・クラス名を許可する。
```

これは正しいです。

しかし、仕様本文である第2章にも実装寄りの表現が混ざっています。

例:

```md
- ベースマップのSVG要素はマウスイベントを一切受け付けず（`pointer-events: none`）...
```

### 問題

`pointer-events: none` は仕様意図の説明としては有用ですが、記法としては CSS 実装詳細です。  
ガイドの「仕様本文では英語識別子・コード値を直接書かない」という原則とはずれます。

### ただし評価は分けるべき

ここは `現状.md` や `開発ガイド.md` ほど深刻ではありません。  
理由は、要件定義書の全体構成はまだ保たれているからです。

問題は「完全に崩れている」ことではなく、「運用が少しずつ侵食されている」ことです。

### 改善方針

仕様本文では以下のように書く方がよいです。

- 実装記法: 書かない
- ユーザー観測可能な要件: 書く

たとえば:

- 現行: `pointer-events: none`
- 改善: ベースマップは地物操作やパン操作を妨げてはならない

実装補遺側でのみ具体的手段を書く形が望ましいです。

## 5. 文書ごとの詳細評価

## 5.1 `指示.md`

### 良い点

- 「根拠となるソースコードを直接引用してください」は非常に良い
- 不確実性の明示を求めている点も良い
- 実装上の失敗や仕様追加時の文書更新を要求している点も良い

### 問題点

- `現状.md` の更新要求が `開発ガイド.md` と衝突している
- 「実装コミットごとに更新」は運用としては正しいが、現状の `現状.md` 構造では持続不可能

### 改善案

- `現状.md` 更新ルールは `開発ガイド.md` に一本化し、`指示.md` は参照だけにする
- あるいは `指示.md` を最上位にし、`開発ガイド.md` 側の矛盾記述を削除する

---

## 5.2 `技術方針.md`

### 良い点

- 技術選定理由は比較的わかりやすい
- DDD 4層、XState、Electron、JSON 保存などの大枠は掴みやすい

### 問題点

- DI など、一部が実装現況とズレている
- LabelRenderer など、責務図が古い
- 200行以下方針が現実と乖離したまま維持されている

### 改善案

- 「現状」と「理想」を分離する
- 実装差分は明示的に「未達事項」として書く

---

## 5.3 `開発ガイド.md`

### 良い点

- 設計原則と注意事項は多くが妥当
- 文書とコードの整合を重視する姿勢は良い

### 問題点

- `useMachine` 記述が古い
- `現状.md` ルールが `指示.md` と衝突
- 守られていない規約が整理されず放置されている

### 改善案

- 実装済みの作法へアップデートする
- 文書運用ルールは1箇所を正本にする
- 「理想だが未達」の規約には注記を付ける

---

## 5.4 `現状.md`

### 良い点

- 情報量は非常に多い
- テスト数、E2E 状況、フェーズ進行などの情報源としては価値がある

### 問題点

- 役割が多すぎる
- 自己矛盾がある
- 完了の定義が曖昧
- Electron 実検証とブラウザ E2E が区別されていない

### 改善案

- 役割分割が必要
- 最低でも「実装済み」「残課題」「既知不具合」「履歴」を章ではなく別文書へ切るべき

---

## 5.5 `要件定義書.md`

### 良い点

- 冒頭の分離方針は良い
- 機能要件の網羅性は高い
- 実装補遺としての第4章・第5章は用途が明確

### 問題点

- 本文に実装ディテールが少し侵入している
- 仕様本文としてはやや技術寄りの表現が混ざる箇所がある

### 改善案

- 本文はよりユーザー観測可能な振る舞いへ寄せる
- 実装補遺への退避を徹底する

## 6. 根本原因の整理

文書の質が落ちている根本原因は、個別の書き方よりも、次の構造問題です。

### 6.1 正本が定義されていない

同じ話題が複数文書に分散し、それぞれが部分的に更新されています。

例:

- `現状.md`
- `開発ガイド.md`
- `指示.md`

が、文書運用ルールそのものを別々に持っています。

### 6.2 「事実」と「方針」と「希望」が同じ口調で書かれている

たとえば技術方針の DI 記述は、現状説明にも理想像にも読めます。  
この曖昧さが誤読を生みます。

## 9. 最終評価

今のドキュメント群は、内容の薄さが問題なのではありません。
