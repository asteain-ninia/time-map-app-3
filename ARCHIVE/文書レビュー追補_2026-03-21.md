# 文書レビュー追補 2026-03-21

## 前提

- この追補は、`time-map-app-2` の実装遺産とのズレを問題視するものではない。
- `time-map-app-3` を新規再設計として見たときに、始動文書として修正した方がよい点だけを整理する。
- 既存の `文書レビュー.md` / `文書レビュー返信.md` は上書きせず、本書を追加レビューとして扱う。

## 重要度高

### 1. 文書間で正典の責務名とUI骨格が揺れている

#### 問題

現状の文書群では、同じ責務に対して別の名前・別の分割が併記されている。

- `開発ガイド.md` では、アプリケーション層として `UpdateFeatureUseCase` を列挙している。
- `要件定義書.md` では、同じ位置に `UpdateFeatureAnchorUseCase` を列挙している。
- `要件定義書.md` では `ResolveConflictUseCase` を列挙している。
- `開発ガイド.md` では `ResolveFeatureAnchorConflictsUseCase` を列挙している。
- `開発ガイド.md` では UI を `MapView` / `TimelineView` / `ToolbarView` / `SidebarView` としている。
- `要件定義書.md` では UI を `MapCanvas` / `TimelinePanel` / `Toolbar` / `Sidebar` としている。
- `開発ガイド.md` では `MapViewModel` や `ViewModel` を前提にした記述が残っている一方、`要件定義書.md` では Svelte コンポーネント + XState ステートマシンが前面に出ている。

#### 何がまずいか

実装者が文書を素直に読むと、どの命名と責務分割を正とすべきか判断できない。
その結果、AI ごとに別の骨格で実装が始まり、初期段階から設計の分岐が起きる。

#### 推奨対応

- 要件定義書・技術方針・開発ガイドの3文書で、責務名を完全に統一すること。
- 「要件定義書の第5章を実装上の正典とする」など、正典の所在を明記すること。
- `ViewModel` を採用するのか、Svelte + XState に集約するのかを先に決め、混在させないこと。

## 重要度高

### 2. 初期実装フェーズの文書に保守フェーズ用ルールが混在している

#### 問題

`指示.md` には、初期実装フェーズ・不具合修正フェーズ・リファクタリングフェーズが同居している。
また、`開発ガイド.md` 冒頭にも保守向けの強い制約が含まれている。

例:

- 「余計な変更は行わないこと」
- 「変更箇所を最小限にとどめること」
- 「既存のコメントは温存すること」
- 「まずは原因の解析に注力すること」
- `現状.md` の運用ルール

さらに、`指示.md` は不具合修正フェーズで `現状.md` を参照させているが、`time-map-app-3` にはそのファイルがまだ存在しない。

#### 何がまずいか

新規立ち上げで必要なのは、保守の慎重さよりも「今このフェーズで読むべき文書の明確化」である。
同じファイル内に非アクティブなルールを残すと、読まないでほしい記述もコンテキストとして流入する。

#### 推奨対応

- 保守用文書はアクティブ文書から物理的に分離すること。
- `AGENTS.md` から参照する入口を、当面は初期実装用文書のみに限定すること。
- 保守用文書は `DONT_README/` あるいは `ARCHIVE/保守ルール/` のような別ディレクトリへ封印し、初期実装フェーズでは参照対象から外すこと。

#### 推奨ディレクトリ例

```text
AGENTS.md
ACTIVE_DOCS/
  初期実装/
    指示.md
    技術方針.md
    開発ガイド.md
DONT_README/
  保守フェーズ/
    指示.md
    開発ガイド.md
    現状運用.md
ARCHIVE/
  旧レビュー類
```

#### 運用上の要点

- 読ませたくない文書を同じファイル内に残さないこと。
- フェーズ切替時は本文を混在させず、`AGENTS.md` の参照先だけを切り替えること。

## 重要度中

### 3. 初期実装向け指示としては、推論禁止が強すぎる

#### 問題

`指示.md` には次のような記述がある。

- 「論ずるときは、根拠となるソースコードを直接引用してください。」
- 「引用したコードを基に確信を持って議論をしてください。」
- 「可能性がある等といった不確実な推論をしてはいけません。」

#### 何がまずいか

新規実装フェーズでは、まだコードが存在しない領域や、仕様上これから決めるべき空白が必ず出る。
その局面で不確実性の明示まで禁止すると、実装者は仕様上の判断保留や前提確認を書きにくくなる。
結果として、曖昧さが消えるのではなく、黙って埋められる。

#### 推奨対応

初期実装用の指示では、次のように弱めた方がよい。

- 既存コードがある論点では、根拠コードを引用すること。
- 既存コードがない論点では、要件定義書・技術方針・開発ガイドのどれを根拠にしたか明示すること。
- 不確実性がある場合は、その不確実性を明示したうえで保守的に進めること。

## 重要度中

### 4. XState に関する表現が強すぎる

#### 問題

`技術方針.md` と `開発ガイド.md` では、XState によって「不正な状態遷移をコンパイル時に防止」と読める表現になっている。

#### 何がまずいか

XState の強みは、状態遷移を宣言的に記述し、実行時の振る舞いと設計意図を明確にすることにある。
ここを「コンパイル時に防止」と言い切ると、型検査で保証される範囲と、実行時マシンで保証される範囲が混同される。

#### 推奨対応

次のような表現に留めるのが妥当。

- 状態遷移を宣言的に定義し、不正遷移を起こしにくくする。
- 状態遷移図を明示することで、仕様と実装の対応を確認しやすくする。
- 型定義と組み合わせることで、一部のイベント・コンテキスト不整合を事前に検出しやすくする。

## 補足

### 良い点

今回の文書群には、修正ではなく維持したい強みもある。

- `FeatureAnchor` を中核に据えた時間モデルは明確である。
- リング階層と飛び地・穴の扱いは、要件としてかなり具体化されている。
- 競合解決を `Prepare / Resolve / Commit` に分ける設計は、複雑な編集処理を整理しやすい。
- 技術方針は、採用技術の理由づけ自体は概ね筋が通っている。

## 先に直す順序

3. 初期実装用 `指示.md` の文言を、新規開発向けに調整
4. XState など、保証範囲を言い過ぎている記述の補正
