# 地物時空変遷編纂機 - 開発前文書レビュー

レビュー対象: 要件定義書.md、技術方針.md、開発ガイド.md、指示.md

## 総合評価

4文書全体として、開発開始に十分な品質と情報量を備えている。ドメインモデル（FeatureAnchor＝歴史の錨という概念設計）、技術スタック選定の根拠記述、DDDの層分離方針、開発ガイドの「落とし穴」セクションなど、いずれも丁寧に練られている。

以下に検出した問題・改善案・追加推奨事項を分類して記載する。

---

## A. 修正すべき問題（不整合・欠落）

### A-1. セクション番号の重複（要件定義書）

- **箇所**: 要件定義書.md 846行目と879行目
- **内容**: 「2.6.1 設定項目一覧」と「2.6.1 アプリ設定」が同一番号で存在する。
- **対処**: 後者のアプリ設定に別の番号（2.6.4等）を振る。

### A-2. 保存形式例とデータモデルの不整合（要件定義書）

- **箇所**: 要件定義書.md 1051-1055行目（データモデル）、1141-1145行目（保存形式例）
- **内容**:
  - データモデル（4.1節）では `property.style` 配下に `fillColor`, `selectedFillColor`, `autoColor`, `palette` を定義している。
  - 一方、保存形式例（4.2節）のJSON例では `property.attributes.styleOverrides` 配下に `fill`, `stroke`, `strokeWidth` を使用している。
  - フィールド名（`fillColor` vs `fill`）、格納場所（`style` vs `attributes.styleOverrides`）、フィールド構成（`autoColor`/`palette` が例に無い、`stroke`/`strokeWidth` がモデルに無い）のいずれも一致していない。
- **対処**: 保存形式例をデータモデルの定義に合わせて修正する。`stroke`/`strokeWidth` が必要であればデータモデル側にも追加する。

### A-3. 上位領域不在時の `parentId` 表現が不統一（要件定義書）

- **箇所**: 要件定義書.md 1063行目（データモデル）、1167行目（保存形式例）
- **内容**:
  - データモデルでは `parentId` はオプショナル（`parentId`?: 文字列）と定義されている。
  - 保存形式例では `"parentId": "0"` と記載されている。
  - `"0"` はマジック値であり、「上位領域なし」を表すなら `null` または省略が適切。
- **対処**: 保存形式例を `"parentId": null` または `parentId` フィールド自体を省略する形に修正する。

### A-4. 共有頂点グループのデータモデルが未定義（要件定義書）

- **箇所**: 要件定義書.md 174-196行目（仕様本文）、4.1節（データモデル全体）
- **内容**:
  - 仕様本文では共有頂点システムを詳細に記述している（近接検出、共有化、連動移動、共有解除など）。
  - しかし第4章のデータモデルに、共有頂点関係を永続化するための構造（例: 共有頂点グループのID、メンバー頂点IDリストなど）が定義されていない。
  - 現状のデータモデルでは、共有関係の保存・復元ができない。
- **対処**: 第4章に共有頂点グループのデータ構造を追加する。例:
  ```
  SharedVertexGroup（共有頂点グループ）
    - id: 文字列。グループの一意ID。
    - vertexIds: 文字列の配列。共有関係にある頂点IDのリスト。
    - representativeCoordinate: Coordinate。代表座標。
  ```
  あわせて保存形式例にも `sharedVertexGroups` 配列を追加する。

### A-5. イベントマーカーのデータモデルが未定義（要件定義書）

- **箇所**: 要件定義書.md 374行目（仕様本文）、4.1節（データモデル全体）
- **内容**: タイムライン上にユーザーが設定できる「イベントマーカー」に言及しているが、第4章にその永続化構造が存在しない。
- **対処**: 第4章にイベントマーカーのデータ構造を追加する。例:
  ```
  TimelineMarker（タイムラインマーカー）
    - id: 文字列。マーカーの一意ID。
    - time: TimePoint。マーカーが示す時刻。
    - label: 文字列。マーカーの表示名。
    - description?: 文字列。補足説明。
  ```

### A-6. 仕様本文内の英語識別子の混在（要件定義書）

- **箇所**: 要件定義書.md 第0章（方針宣言）、2.2節周辺（本文）
- **内容**:
  - 第0章で「仕様本文では英語識別子・内部キー名・クラス名・コード値を直接記載しないこと」と宣言している。
  - しかし2.2節の仕様本文に `TimePoint`、`FeatureAnchor`、`RGBA` などの英語識別子が散見される。
  - 重大度は低いが、自ら定めた方針との不一致。
- **対処**: 仕様本文内の英語識別子を日本語表現に置換する（例: `TimePoint` → 「時間点」、`FeatureAnchor` → 「歴史の錨」、`RGBA` → 「透過度を含む色指定」）。既に用語集で日本語名が定義されているものはそちらに統一する。

## B. 改善推奨

### B-1. 編集機能の分類位置が不適切（要件定義書）

- **箇所**: 要件定義書.md 222-275行目
- **内容**: 「分裂と割譲/合邦機能」「合体（結合）機能」が「2.1 地図表示機能」の配下に置かれている。これらは表示機能ではなく編集機能であり、読み手が機能を探す際に混乱を招く。
- **提案**: 2.3節（オブジェクト編集機能）の配下に移動するか、独立した「2.X 地物操作機能」セクションを新設する。

### B-2. 用語集「合体（結合）」の定義が仕様本文と不一致（要件定義書）

- **箇所**: 要件定義書.md 1394行目（用語集）、252行目（仕様本文）
- **内容**:
  - 用語集: 「隣接する**2つ**の**最下位**面情報を一つの面情報にまとめる操作」
  - 仕様本文: 「同じレイヤーに属する**複数の**面情報を…1つの新しい面情報に統合する操作」、制約は「下位領域を持たないこと」
  - 「2つ」と「複数」、「最下位」と「下位領域を持たない」がそれぞれ一致していない。
- **提案**: 用語集の記述を仕様本文に合わせて修正する。

### B-3. SvelteおよびXStateのバージョンが未指定（技術方針）

- **箇所**: 技術方針.md 15-30行目
- **内容**: Svelte 4と5ではAPI体系が大幅に異なる（stores vs runes）。XState v4とv5も同様。バージョン未指定のまま開発を始めると、参照する情報やコードパターンにブレが生じる。
- **提案**: 使用バージョンを明記する（2026年3月時点ではSvelte 5 + XState v5が妥当と思われる）。

### B-4. Electron + Vite統合の構成指針が未記載（技術方針）

- **箇所**: 技術方針.md 41-43行目
- **内容**: Electron + Vite + Svelteの組み合わせは、メインプロセス・プリロード・レンダラーの3エントリポイントに対するビルド構成が必要になる。electron-vite等のツール選定や、プロセス間通信（IPC）のパターンに触れておくと、実装時の迷いが減る。
- **提案**: 技術方針にElectron固有のビルド構成方針（使用ツール、プロセス分離パターン）を追記する。

### B-5. 座標系の定義が不明確（要件定義書）

- **箇所**: 要件定義書.md 1031-1033行目（Vertexデータモデル）
- **内容**: Vertexは `x`（数値）と `y`（数値）を持つが、正距円筒図法における座標系との対応関係が明記されていない。具体的には:
  - `x` は経度、`y` は緯度に対応するのか
  - 値域は -180〜180 / -90〜90 か、0〜360 / 0〜180 か
  - 横方向無限スクロール（仕様本文2.1節）との座標ラッピングはどう扱うか
- **提案**: 第4章のVertex定義に座標系の規約を追記する。

### B-6. メタデータの保存形式例に設定項目が不足（要件定義書）

- **箇所**: 要件定義書.md 1206-1218行目（metadata例）、848-863行目（設定項目一覧）
- **内容**: 設定項目一覧に定義されている以下の項目が、保存形式例の `metadata.settings` に反映されていない:
  - 扁平率
  - 面情報ラベル画面占有率閾値
  - 新規面の自動配色既定値
  - 既定パレットテーマ
  - カスタムパレット
- **提案**: 保存形式例を設定項目一覧と整合させる。

### B-7. ファイル行数目安の整理（開発ガイド・技術方針）

- **箇所**: 技術方針.md 18行目、開発ガイド.md 91行目・120行目
- **内容**:
  - 技術方針: Svelteコンポーネントは「200行以下」
  - 開発ガイド 2.1節: 一般ファイルは「300行を目安」
  - 開発ガイド 2.5節: Svelteコンポーネントは「200行以下」
  - 矛盾はないが、開発ガイド 2.1節の300行基準にSvelteコンポーネントの200行基準が併記されていない。
- **提案**: 開発ガイド 2.1節に「Svelteコンポーネントは200行以下を目安（2.5節参照）」と補記する。

## C. 追加を推奨する項目

### C-1. 初期実装の優先順位・フェーズ計画

- **理由**: 要件が非常に多く、全てを一度に実装するのは非現実的。開発の順序を誤ると、基盤が不安定なまま高度な機能を積み上げることになる。
- **提案**: MVP（最小限の動作するプロダクト）とフェーズ分割を定義する文書を追加する。例:
  - Phase 1: プロジェクト基盤（Electron + Vite + Svelte構成）、基本地図表示（SVG描画、ズーム/パン）、レイヤー管理、面情報の追加と表示
  - Phase 2: 時間管理（FeatureAnchor、タイムラインUI、時間に応じた表示切替）
  - Phase 3: 編集機能（頂点編集、共有頂点、排他性チェック、エッジ滑り）
  - Phase 4: 高度な編集（分割、結合、競合解決プロセス）
  - Phase 5: 補助機能（測量モード、自動配色、検索、詳細文書）
  - Phase 6: 堅牢化（自動保存、バックアップ、データ整合性チェック、パフォーマンス最適化）

### C-2. キーボードショートカットの一覧表

- **理由**: ショートカットが各機能の説明に分散しており、全体像が把握しにくい。
- **提案**: 要件定義書にショートカット一覧セクションを追加する。現在散在しているものの例:
  - Ctrl+Z: アンドゥ
  - Ctrl+Y / Ctrl+Shift+Z: リドゥ
  - Ctrl+S: 保存
  - Ctrl+O: 読み込み
  - Delete: 選択要素の削除
  - Escape: 操作キャンセル / 選択解除
  - Shift+クリック: 頂点の追加選択/除去

### C-3. `Feature.properties` フィールドの保存方針の明確化

- **箇所**: 要件定義書.md 1080行目
- **内容**: `Feature` の `properties: PropertyProjectionの配列` は「表示互換のための読み取り専用投影」と説明されている。これが保存対象なのか、`anchors` から毎回算出されるのかが不明確。
- **提案**: 以下のいずれかを明記する:
  - (a) 保存しない（anchorsから都度算出） → データモデルから除外するか「保存対象外」と注記
  - (b) 保存する → 保存形式例に反映し、anchorsとの同期方針を定義

## D. あるべきでない記述

### D-1. 指示.mdの内容が開発開始フェーズに適合していない

- **箇所**: 指示.md 全体
- **内容**: 指示.mdの内容は「現状.mdに記載されている不具合を修正する」ことを前提としたAIアシスタントへの指示である。開発開始時点では現状.mdは存在せず、修正すべき不具合もない。
- **提案**: 以下のいずれかの対処を行う:
  - (a) 現在の内容を「不具合修正時の指示テンプレート」としてリネーム（例: 指示_不具合修正.md）し、開発開始フェーズ向けの指示を別途作成する
  - (b) 指示.mdに開発フェーズごとのモード切替（初期実装 / 不具合修正 / リファクタリング等）を追記する
