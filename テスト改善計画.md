# テスト改善計画

> 2026-04-09 作成。テスト品質の底上げとエッジケースカバレッジの強化を目的とする。

## 0. 方針

### 0.1 カバレッジ目標

業界のベストプラクティス（Google Engineering Practices, Martin Fowler "TestCoverage"）を参考に、
レイヤー別に目標を設定する。100%は保守コスト対効果が悪いため目指さない。

| レイヤー | 目標 | 根拠 |
| --- | --- | --- |
| ドメイン層（entities / value-objects / services） | 90%+ | 純粋ロジック。副作用なし。最もテストしやすく、バグの影響が大きい |
| アプリケーション層（use cases / commands / queries） | 85%+ | ビジネスフロー。ドメイン層を組み合わせるため結合テスト的性質 |
| インフラ層（rendering / persistence / config） | 75%+ | 外部依存あり。間接カバレッジ（上位テスト経由）を含めて評価 |
| プレゼンテーション層（utils / state machines） | 80%+ | 純粋関数部分はユニット、Svelteコンポーネントは E2E で補完 |
| Electron / Preload | 60%+ | エントリポイントは E2E 依存。DI化済みハンドラはユニットで検証 |

### 0.2 テスト追加の方法

- **追加予定のテスト**: `it.todo('説明')` で宣言し、テストスイートを green のまま維持
- **バグを発見したテスト**: テストを書いて失敗を確認 → `it.skip` にして `現状.md` に不具合登録
- **正常に通るテスト**: そのままコミット

### 0.3 テストの質についての指針

テストは「数」よりも「検出力」が重要。以下のパターンを優先する:

1. **境界値**: 0, 1, 上限, 上限+1, 負数, NaN, Infinity
2. **空/null**: 空配列, 空Map, null/undefined, 空文字列
3. **型の網羅**: union型（Point/LineString/Polygon）の全分岐
4. **状態遷移**: 初期状態, 遷移直後, 連続操作, 不正遷移
5. **相互作用**: 複数機能の組み合わせ（例: 共有頂点 + ポリゴン検証）
