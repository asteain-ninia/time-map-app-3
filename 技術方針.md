# 地物時空変遷編纂機 - 技術方針

## 1. 技術スタック

### 1.1 デスクトップフレームワーク: Electron
- ファイルシステムへの直接アクセスが必要
- オフライン動作が前提
- クロスプラットフォーム対応（Windows, macOS, Linux）

### 1.2 言語: TypeScript (strict mode)
- ドメインモデルの複雑さ（アンカー、共有頂点、ポリゴンリング構造）に対し、型安全性が不可欠
- コンパイル時にデータ構造の不整合を検出
- インターフェースによるコンポーネント間の契約を明示化

### 1.3 UIフレームワーク: Svelte 5
- **バージョン: Svelte 5**（Runes API）を使用。Svelte 4以前のstores APIは使用しない。
- Virtual DOMなしで軽量 — Electron環境での描画パフォーマンスに有利
- コンポーネント単位でCSSが閉じる — グローバルCSS汚染を防止
- コンポーネント分割が自然で、各ファイルを200行以下に保ちやすい
- `$state`, `$derived`, `$effect` によるリアクティブ宣言

### 1.4 状態管理: XState v5
- **バージョン: XState v5** を使用。v4とはAPI体系が異なるため、v4のドキュメント・コード例を参照しないこと。
- ツール状態（選択・頂点移動・分割・ホール作成等）を明示的な状態機械として表現
- 不正な状態遷移をコンパイル時に防止
- 状態遷移図を自動生成でき、仕様とコードの一致を視覚的に確認可能
- 主な状態遷移:
  ```
  idle → selecting → dragging → snapping → idle
       → drawing → placing_vertex → closing_polygon → idle
       → splitting → drawing_cut_line → confirming → idle
  ```

### 1.5 レンダリング: カスタムSVG
- SVG標準のviewBox属性による座標変換を活用
- レンダリング層を責務ごとに分割:
  - ViewportManager: 座標変換
  - FeatureRenderer: 形状描画
  - StyleResolver: スタイル計算
  - LabelRenderer: ラベル配置
- polygon-clippingライブラリによるブーリアン演算

### 1.6 ビルドツール: Vite (electron-vite)
- **electron-vite** を使用し、Electron固有の3エントリポイント（メインプロセス・プリロード・レンダラー）のビルドを統合管理する。
- TypeScript + Svelteとのネイティブ統合
- 高速なHMR（ホットモジュールリプレースメント）— レンダラープロセスのみHMR対象
- メインプロセスとプリロードスクリプトはNode.js環境向けにビルドし、レンダラーはブラウザ環境向けにビルドする。
- プロセス間通信（IPC）はプリロードスクリプトで `contextBridge.exposeInMainWorld` を用いて安全に公開し、レンダラーから直接Node.js APIにアクセスしない。

### 1.7 テスト: Vitest + Playwright
- ドメイン層: jsdom不要の純粋な単体テスト（Vitest）
- ジオメトリ計算: カバレッジ100%目標
- UI: Playwright（E2E）で主要シナリオのみ

## 2. アーキテクチャ

### 2.1 ドメイン駆動設計（DDD）4層構造
ドメインの複雑さ（時空間データ、階層的なポリゴン構造、共有頂点の連動、競合解決プロセス）に対して適切な分離を実現する。

```
ドメイン層       ← ビジネスルール・計算・エンティティ
アプリケーション層 ← ユースケース・コマンド/クエリ・履歴管理
インフラストラクチャ層 ← 永続化・レンダリング・設定
プレゼンテーション層  ← Svelteコンポーネント・状態機械・コントローラ
```

### 2.2 依存性注入（DI）コンテナ
- 全サービスの初期化をDIコンテナで一元管理
- テスト時にモック差し替えが容易
- 層間の疎結合を維持

### 2.3 イベント駆動アーキテクチャ
- ドメイン状態の変更はイベントバスで通知
- コンポーネント間の直接参照を避け、疎結合を維持
- 主要イベント: FeatureAdded, TimeChanged, LayerVisibilityChanged 等

## 3. データモデル

### 3.1 時間モデル: FeatureAnchor（歴史の錨）
- 地物の経時的状態を「錨」のリストとして保持
- 各錨は特定時間範囲における地物の完全状態スナップショット
- 任意の時点の状態は、有効なアンカーの検索で取得

### 3.2 空間モデル: リング階層構造
- ポリゴンはリングの階層で定義（領土リング → 穴リング → 飛び地リング → …）
- 穴・飛び地の任意深さのネストに対応
- 共有頂点による境界連動

### 3.3 永続化: JSON形式
- ローカルファイルシステムに保存
- 自動保存（5分間隔）+ 世代管理（最大5世代）
- バージョン管理付きで形式の互換性を管理

## 4. 設計判断の根拠

### 4.1 Electronを選んだ理由
ファイル操作が頻繁でオフライン動作が前提のため、Webアプリやモバイルアプリではなくデスクトップアプリとする。Tauriも候補だが、エコシステムの成熟度からElectronを選択。

### 4.2 Svelteを選んだ理由
地図アプリはDOM操作が頻繁で、Virtual DOMのオーバーヘッドを避けたい。Reactは学習コストが高く、Vueはコンポジション APIの複雑さがある。Svelteはコンパイル時にリアクティブコードを生成し、ランタイムコストが最小。

### 4.3 XStateを選んだ理由
本アプリは多数の編集ツール（選択、頂点編集、分割、穴作成、結合等）を持ち、各ツールの状態遷移が複雑。状態がViewModel/UseCase/Serviceに分散すると「今どのモードにいるか」が追いにくくなる。XStateは状態遷移を明示的に定義し、不正遷移を防ぐ。

### 4.4 TypeScriptを選んだ理由
ドメインモデルが複雑で、コンポーネント間のデータ受け渡しでの型不整合（プレーンオブジェクトとドメインインスタンスの混在、プロパティ名の不一致等）がバグの主因となりやすい。TypeScriptのstrict modeはこれらの問題をコンパイル時に検出する。

### 4.5 レンダリング層分割の理由
レンダリングは座標変換・形状描画・スタイル計算・ラベル配置という異なる責務を持つ。これらを単一クラスにまとめると巨大化する。責務ごとに分割することで、各モジュールを独立してテスト・修正できる。

### 4.6 polygon-clippingを採用した理由
ポリゴンのブーリアン演算（和・差・交差）は自前実装すると膨大な工数とバグリスクを伴う。polygon-clippingは十分に成熟したライブラリであり、穴や飛び地を含む複雑なポリゴンに対応している。
