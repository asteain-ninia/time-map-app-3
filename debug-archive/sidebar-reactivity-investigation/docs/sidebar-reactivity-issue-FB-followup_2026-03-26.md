## 1. 先に結論

今回の追記で、前回より文書の質はかなり上がっています。

特に重要なのは、前回の最大の弱点だった

- `Sidebar` に `selectedFeatureId` が渡っていないのではないか
- `onFeatureSelect` が未接続ではないか

という疑義が、**今回の実装と再現ログでいったん解消された**ことです。

実際、現行コードでは `App.svelte` 側で次のように接続されています。

```svelte
<Sidebar
  {currentTime}
  {features}
  {selectedFeatureId}
  onFeatureSelect={(id) => { selectedFeatureId = id; }}
  {onPropertyChange}
/>
```

したがって、今回の文書は前回より一段深く、

**「親子配線は通っているのに、DOM反映だけが止まって見える」**

というところまで前進しています。

ただし、それでもまだ

**Svelte 5 の本体バグと断定するには早い**

です。

理由は、現在の再現条件がまだ「最小構成」ではなく、切り分け上の重要な確認があと少し残っているからです。

## 2. 今回の進展点

今回の追記で、少なくとも次は確認できています。

### 2.1 親子の接続は成立している

`App.svelte` では `selectedFeatureId` が `$state` として保持されています。

```ts
let selectedFeatureId = $state<string | null>(null);
```

そして `Sidebar` へ明示的に渡されています。

```svelte
<Sidebar
  {currentTime}
  {features}
  {selectedFeatureId}
  onFeatureSelect={(id) => { selectedFeatureId = id; }}
  {onPropertyChange}
/>
```

これは前回の状態と決定的に違います。

### 2.2 callback は親まで届いている

ログ上も、`Sidebar` 側のクリックから `App` 側の更新まで到達しています。

```text
[BROWSER] SIDEBAR_CLICK f-2 prop: f-1
[BROWSER] APP_onFeatureSelect f-2 prev: f-1
[BROWSER] APP_after: f-2
```

したがって、

- クリックイベントが発火していない
- callback が繋がっていない
- 親の代入が走っていない

という線はかなり薄くなりました。

### 2.3 子コンポーネントのイベントハンドラからは更新済み値が見えている

2回目以降のクリックで `SIDEBAR_CLICK ... prop: f-1` や `prop: f-2` が見えているため、少なくとも `Sidebar` インスタンスの JavaScript 実行文脈では、前回更新後の値にアクセスできています。

これは重要な観測です。

---

## 3. それでもまだ断定できない理由

## 3.1 「prop は正しく伝搬している」は少し言いすぎ

文書では次のように整理しています。

```md
1. prop は正しく伝搬している
2. callback は正しく動作
3. しかしテンプレートに反映されない
```

ここで 2 はかなり強く言えます。  
しかし 1 は、まだ少し慎重に書くべきです。

今回言えるのは厳密には次です。

- 子コンポーネントの `onclick` ハンドラ内では、前回更新された `selectedFeatureId` を読める

ここからすぐに

- 親の再描画は正常
- 子の prop 更新スケジューリングは正常
- テンプレート依存追跡だけが壊れている

とまではまだ言えません。

なぜなら、**親DOMが実際に更新されているか** をまだ見ていないからです。

## 3.2 親DOMが更新しているか未確認

現状のログは、親の script 内 state 更新までは示しています。

```text
[BROWSER] APP_after: f-1
```

しかし、まだ不足している確認があります。

たとえば `App.svelte` 側に

```svelte
<p id="app-debug">{selectedFeatureId}</p>
```

を置いて、

- 親の DOM は更新されるのか
- 子の DOM だけが更新されないのか

を分けて見る必要があります。

この差は大きいです。

もし親DOMも更新されないなら、問題は `Sidebar` の prop 伝搬ではなく、もっと上流の再描画やスケジューリングです。  
逆に親DOMだけ更新されるなら、初めて「親→子反映」や「子テンプレート依存追跡」の疑いが強くなります。

## 3.3 「最小構成」という表現はまだ不正確

文書では現状を「現在の実装（最小構成）」と書いています。

しかし実際には、親は約1300行の `App.svelte` のままです。

このコンポーネントは少なくとも以下を同時に抱えています。

- ツール状態
- 地図クリック処理
- Undo/Redo
- バックアップ
- Dirty状態
- 分割・結合
- 競合解決
- タイムライン
- 複数の derived/state

この状態は、切り分け上の「最小構成」ではありません。

今あるのは、

**アプリ内でのかなり絞った再現**

であって、

**Svelte バグ報告に持っていける最小再現**

ではありません。

ここは言葉を正確にした方がよいです。

## 3.4 デバッグテストが依然として契約を固定していない

`e2e/debug-sidebar-reactivity.spec.ts` は今回も最後が次です。

```ts
expect(true).toBe(true);
```

つまり、

- `DEBUG: f-1` になるべき
- item[0] に `selected` class が付くべき

という期待は、まだテストに昇格していません。

この状態だと、調査のたびにログを人間が読んで解釈する必要があります。

デバッグメモとしては十分でも、検証資産としてはまだ弱いです。

## 4. 実装者への再助言

今回の文書を読んだ実装者に、次はこう返します。

## 4.1 前回より進んでいます

これは先に明言してよいです。

前回は「そもそも `Sidebar` に prop を渡していないのでは」という段階でした。  
今回はその疑義を潰し、**親 state 更新と子イベント観測のズレ** まで絞れています。

ここはちゃんと前進です。

## 4.2 次に確認すべきは「親DOMも止まるか、子DOMだけ止まるか」です

今の切り分けで一番価値が高い追加確認はこれです。

1. `App.svelte` に `{selectedFeatureId}` を直接描画する
2. クリック後にその文字列が更新されるか確認する
3. `Sidebar` 側の `DEBUG` と比較する

これで問題領域が大きく狭まります。

## 4.3 本当に最小再現を作ってください

次の 2 コンポーネントだけに切った再現を作るべきです。

- 親: `$state` で `selected` を持ち、ボタンまたは callback で更新
- 子: `$props()` で受けて `<p>{selected}</p>` を描画

この 20〜30 行程度の再現で壊れるなら、初めて Svelte 本体や dev server の問題を強く疑えます。

逆にここで壊れないなら、今のアプリ内にある別の要因です。

## 4.4 assertion を入れてください

たとえば次のように書くべきです。

```ts
await items.nth(0).click();
await expect(debugP).toHaveText('DEBUG: f-1');
await expect(items.nth(0)).toHaveClass(/selected/);
```

これで「人間がログを読む」調査から、「落ちる契約」を持つ検証に進めます。

## 4.5 文書の解釈も少しだけ弱めてください

今の文書の

> prop は正しく伝搬している

は、次のように直す方が正確です。

> 子コンポーネントのイベントハンドラからは、前回更新後の prop 値を読める

これなら観測事実に忠実です。

## 5. 推奨する次の切り分け順

次にやるべき順番は、これがよいです。

1. `App.svelte` 側にも debug 表示を置き、親DOM更新を確認する
2. `Sidebar` 専用の assertion 付き E2E に変える
3. 20〜30 行の親子だけの最小再現を別ファイルで作る
4. dev server を完全再起動して再確認する
5. 必要ならその段階で初めて Svelte 側の issue を疑う

この順にしないと、

- アプリ固有問題
- HMR/開発環境問題
- Svelte ランタイム問題

が混ざったままになります。

## 6. 最終評価

今回の追記で、文書は確実に良くなっています。

特に良いのは、

- 前回の配線漏れ疑義を潰したこと
- ログの粒度が上がったこと
- `onclick` の JS 文脈と DOM 表示のズレに着目できていること

です。

ただし、現時点の最終評価は次です。

- 前回よりかなり良い
- 観測は面白い
- しかしまだ「Svelte 本体バグ」の証明には足りない

今の段階で最も重要なのは、**親DOMと子DOMを分離して観測すること** と、**本当の最小再現を作ること** です。

ここまで行けば、文書は「調査メモ」から「再現報告」に変わります。
