> **フェーズ注記**: 以下の注意事項は主に保守・不具合修正フェーズを想定した制約である。初期実装フェーズでは「変更箇所を最小限にとどめる」「既存コメントの温存」等の制約は必要に応じて緩和される。ただし「根本原因の修正」「省略の禁止」「教訓の記録」は全フェーズで有効。

- 余計な変更は行わないこと。
- 指示されていない変更をしないこと。
- 変更箇所を最小限にとどめること。
- 既存のコメントは温存すること。
- コメントを記載するときは、本質的に必要な内容を書くこと。一時的な情報を含めないこと（「// 修正 :～」「//ここを修正」「//--修正ここから--」などの書き方をしないこと）。
- 論ずるときは、根拠となるソースコードを直接引用すること。
- コードの修正時は、「～も同様に～」といった省略は一切禁止。
- 不具合修正時は、必ず根本原因を修正すること。場当たり的な対応をしないこと。
- まずは原因の解析に注力すること。
- 実装上の失敗は開発ガイドに保存していくこと。
- 実装のやり残しは現状.mdに追記していくこと。
- **文書の役割分担**:
  - `現状.md` = 未解決課題・既知不具合・残作業・テスト総数。完了した課題は原則として即時削除すること。

- 現状.mdから記載を削除する場合、それがリストの最後でない限り欠番として残すこと。
- 要件定義書は、仕様本文（完全日本語）と内部実装補遺を分離して記述すること。

## 1. アーキテクチャ概要

本アプリケーションはドメイン駆動設計（DDD）の原則に基づいて構築される。

### 1.1 技術スタック

| レイヤー                   | 技術                         |
| -------------------------- | ---------------------------- |
| デスクトップフレームワーク | Electron                     |
| 言語                       | TypeScript (strict mode)     |
| UIフレームワーク           | Svelte 5 (Runes API)         |
| 状態機械                   | XState v5                    |
| レンダリング               | SVG                          |
| データ形式                 | JSON（ローカルファイル保存） |
| ビルド                     | Vite (electron-vite)         |
| テスト                     | Vitest + Playwright          |
| 幾何学計算                 | polygon-clipping             |

### 1.2 ドメイン層

アプリケーションの中核となるビジネスロジック。

- エンティティ: Vertex, Feature, Point, Line, Polygon, Layer
- 値オブジェクト: Coordinate, TimePoint, FeatureAnchor, PropertyProjection, Ring
- ドメインサービス: GeometryService, TimeService, LayerService
- 集約ルート: World

### 1.3 アプリケーション層

ユーザー操作をドメインロジックに変換し、結果を表示層に返す。

- ユースケース: EditFeatureUseCase (facade), AddFeatureUseCase, UpdateFeatureAnchorUseCase, DeleteFeatureUseCase, VertexEditUseCase, NavigateTimeUseCase, ManageLayersUseCase
- PolygonEditService: ポリゴンのリング構造編集
- PrepareFeatureAnchorEditUseCase, ResolveFeatureAnchorConflictsUseCase, CommitFeatureAnchorEditUseCase
- CQRSの原則: コマンド/クエリ/イベントの分離

### 1.4 インフラストラクチャ層

外部リソースへのアクセスやフレームワーク統合。

- JSONWorldRepository, FileSystem, JSONSerializer
- SVGRenderer (split into ViewportManager, FeatureRenderer, StyleResolver, LabelRenderer)
- Logger, ConfigManager

### 1.5 プレゼンテーション層

ユーザーインターフェースとユーザーとの対話。Svelteコンポーネントを使用。

- コンポーネント: MapCanvas, TimelinePanel, Toolbar, Sidebar, ConflictResolutionDialog
- 状態管理: XState v5 ステートマシン (toolMachine, conflictResolutionMachine, timelineMachine)
- XState v5 は `createActor` ベースで手動統合する（`@xstate/svelte` はSvelte 5未対応のため不採用）。ユーザー操作はイベントとしてマシンに送信する

## 2. 実装ガイドライン

### 2.1 一般的なコーディング規約

- TypeScript strict mode を有効にし、any を禁止
- インデントは2スペース
- セミコロン使用
- 命名規則:
  - クラス名: PascalCase
  - 変数/関数: camelCase
  - 定数: UPPER_SNAKE_CASE
  - プライベートメンバー: TypeScriptのprivate修飾子を使用（アンダースコア接頭辞ではなく）
- 関数は1つのことだけを行い、30行以下に抑える
- ネストを3レベル以下に抑える
- コメントは日本語で記述
- 1ファイルあたり300行を目安に分割（Svelteコンポーネントは200行以下、2.5節参照）

### 2.2 ドメイン層の実装ガイドライン

- 不変性: エンティティと値オブジェクトはreadonlyプロパティで設計。状態変更は新インスタンスを返す。
- 値オブジェクトの等価性: equals()メソッドによる属性値比較。
- ドメインサービスはステートレス。
- ドメインロジックは他の層に漏れ出さない。
- 集約ルートの整合性を保つ。集約間参照はIDで。

### 2.3 アプリケーション層の実装ガイドライン

- CQRS原則に従い、コマンドとクエリを分離。
- ユースケースは特定のタスクに集中。
- トランザクション管理（操作の原子性保証）。
- イベント駆動で状態変更を通知。

### 2.4 インフラストラクチャ層の実装ガイドライン

- 外部依存性はこの層に閉じ込める。
- アダプタパターンで抽象化。
- リポジトリは永続化詳細を隠蔽。
- 低レベル例外はアプリケーション固有の例外に変換。

### 2.5 プレゼンテーション層の実装ガイドライン

- Svelteコンポーネントは表示とイベント受付に専念。
- ビジネスロジックはコンポーネントに書かない。ユースケースに委譲。
- XState v5の状態機械でツール状態を管理。状態遷移を宣言的に定義し、不正な遷移を起こしにくくする。
- コンポーネントは200行以下を目安に分割。

### 2.6 テスト戦略

- ドメイン層: jsdom不要の純粋な単体テスト（Vitest）。GeometryServiceはカバレッジ100%目標。
- アプリケーション層: DIで依存を差し替えてテスト。
- UI: Playwright（E2E）で主要シナリオのみ。
- テストは tests/ 配下に、ソースと同じディレクトリ構造で配置（例: `tests/domain/entities/Feature.test.ts`）。
- **テストは実装と同時に書く。** 実装後にテストを後回しにしない。
- 現状.md の実装済みテーブルにはテスト有無カラムを維持し、件数を記録する。

### 2.7 ビルド・テスト環境の注意点

- `vitest.config.ts` にパスエイリアス（`@domain`, `@application` 等）を定義する必要がある。`electron.vite.config.ts` のエイリアスと二重管理になるため、片方を変更したらもう片方も追従すること。

## 6. 開発上の注意点とよくある落とし穴

ここからが最も重要なセクション。過去の開発で得られた教訓を記録。

### 6.1 状態管理と更新ロジック

#### 6.1.1 状態間の依存関係の更新漏れ

- 教訓: ある状態が変化したら、それに依存する他の状態も検証・更新する。例えば「時間が変わったら、選択中の地物がその時間に存在するか検証し、存在しなければ選択解除」。暗黙的ルールはコードで明示的にチェックする。

#### 6.1.2 非表示要素の選択・操作の混入

- 教訓: ユーザー操作の対象は「現在UI上に表示されている要素」に限定する。データソース全体を直接検索せず、Svelteのリアクティブ状態（`$derived`等）が管理する表示状態を基準にする。

#### 6.1.3 Undo/Redoスタック操作の副作用

- 教訓: pushUndoがclearRedoStackを呼ぶような副作用を持つメソッドは、全呼び出しコンテキストで副作用が適切か慎重に検討する。Redo操作からのpushUndoではRedoスタックをクリアしてはならない。

#### 6.1.4 コンポーネント間での状態同期漏れ

- 教訓: 共有リソースを更新したら、依存する全コンポーネントに変更を通知する。イベントバスで汎用的な状態変更イベントを発行し、各コンポーネントがリフレッシュするパターンが有効。リアクティブ状態やXStateコンテキストは内部データが常に最新とは限らない前提で設計する。

#### 6.1.5 プロジェクト設定とアプリ設定の責務混在

- 教訓: プロジェクト固有設定とアプリ全体設定の境界を先に定義し、保存先と反映先を一貫させる。

### 6.2 UIインタラクションとユーザー操作フロー

#### 6.2.1 操作フローの完了トリガー欠落

- 教訓: UIインタラクションのフロー（開始・途中操作・完了/キャンセル）では、必ず「完了」のトリガーを定義・実装する。ユースケースにメソッドがあってもUIからの呼び出しがなければ機能しない。

#### 6.2.2 排他的イベント処理による意図しない動作

- 教訓: 排他的な条件分岐が他の重要な処理をスキップさせていないか確認する。頂点クリック時にその地物の選択も行うべきか等、選択優先順位と同時/排他選択を明確に定義する。

#### 6.2.3 オーバーレイ上のモーダルUIでのイベント競合

- 教訓: 地図の上にモーダルを表示する場合、イベント発生源を検証し、モーダル内のイベントが地図操作をトリガーしないようにする。stopPropagationだけでは不十分な場合があり、リスナー内でevent.targetを判定する。

#### 6.2.4 選択ハイライトで穴が塗り潰される

- 教訓: 穴や飛び地を持つポリゴンの塗りでは、領土リングと穴リングを同じpathのサブパスとして定義し、fill-rule="evenodd"を適用する。本体描画とオーバーレイ描画でリング階層の解釈を共有する。

#### 6.2.5 選択状態の分類漏れ

- 教訓: 複数の選択状態（単一・複数・不明・空）を扱うUIでは、派生済みのヘルパー結果だけでなく、生のIDセットを全て評価して分岐を網羅する。仕様のメッセージとViewの条件分岐を一対一で対応付ける。

#### 6.2.6 設定フォームのスクロール領域不足

- 教訓: フォームの操作ボタンが下部に配置されるUIでは、親の高さとoverflowを明確にし、min-height: 0とoverflow-y: autoを設定する。最小ウィンドウサイズでボタンが見切れていないか確認する。

#### 6.2.7 マウスイベントシーケンスの状態管理不備

- 教訓: mousedown→mousemove→mouseupのシーケンスで、全てのパス（クリック、ドラッグ）で状態が正しく開始・更新・終了されることを保証する。mousedownで設定した状態はmouseupで必ず解除する。mousemoveが発生しなかった場合（クリック）の考慮漏れに注意。

#### 6.2.8 一時的なインタラクション状態と描画の連携不備

- 教訓: 複数ステップのUIインタラクションでは、一時的な状態をXStateマシンのコンテキストで一元管理し、プレビュー描画はそのコンテキストから状態を取得して反映する。

#### 6.2.9 条件付き描画内の未定義参照による更新停止

- 教訓: 選択時やモード切替時にだけ描画されるUI分岐では、渡すprop名とローカル変数名の不一致が `ReferenceError` を起こし、無関係なリアクティブ更新停止に見えることがある。条件付きで出現するツールバーやオーバーレイは、表示契機を含むE2Eで実際に描画させ、ブラウザ実行時エラーも検出する。

#### 6.2.10 横方向無限スクロールでオーバーレイだけが追随しない

- 教訓: 横方向無限スクロールでは、ベース地図や地物本体だけでなく、頂点ハンドル・描画プレビュー・測量線のようなSVGオーバーレイも同じ `wrapOffset` 描画文脈に置く。地図本体とオーバーレイでラップ位置が分離すると、複製地図上に操作対象が出ず、パン後に「表示されない」「ずれて見える」不具合になる。

#### 6.2.11 東西端またぎの描画と無限スクロール本体は別々に担保する

- 教訓: 頂点や測量点の保存座標は `-180〜180` に正規化してよいが、`LineString` / `Polygon` / 大円パスの描画・ヒットテストでは、点列を前後関係からアンラップして `179° → -179°` を短い経路として扱う。同時に `wrapOffset` は `-360 / 0 / +360` の固定列挙ではなく、現在の `viewBox` と交差する全タイルを動的に生成する。保存用正規化座標をそのままパス化したり、隣1枚だけを描画すると、東西端またぎが世界横断線になり、1周以上のパンで空白が出る。

### 6.3 データ整合性と型

#### 6.3.1 ドメインオブジェクトの代わりにリテラルオブジェクトを使用

- 教訓: DDDでは、エンティティや値オブジェクトは定義されたクラスのインスタンスとして扱う。プレーンオブジェクトを渡さない。TypeScriptの型システムでこれを強制する。

#### 6.3.2 コンポーネント間のデータ構造不一致

- 教訓: コンポーネント間でデータを渡す際、プロパティ名・型・構造が受け取り側の期待と一致しているか確認する。TypeScriptのインターフェースで契約を明示する。

#### 6.3.3 コンテキスト上の時刻とシステム時刻の混同

- 教訓: 時間軸を持つアプリでは、データのタイムスタンプは必ずコンテキスト上の時刻（タイムラインの現在時刻）を参照する。時刻源を一元管理するサービスから取得する。

#### 6.3.4 継承関係におけるメソッドオーバーライド漏れ

- 教訓: クラス継承でサブクラスが追加状態を持つ場合、基底クラスのwithXxx系メソッドはサブクラスでオーバーライドが必要か慎重に検討する。

#### 6.3.5 Undo/Redoのデータ復元

- 教訓:
  - 「操作を元に戻す」に必要な情報を正確に特定し、依存データも含めて過不足なく保存する。
  - 履歴にはプレーンオブジェクトを保存し、復元時は必ず新しいドメインインスタンスを生成する。
  - デシリアライズは再帰的に行い、ネストされた値オブジェクトも正しく変換する。
  - 履歴データのフォーマットは一貫させる。

#### 6.3.6 データ構造と描画ロジックの不整合

- 教訓: データ構造の全有効状態組み合わせを、描画・シリアライズ・操作の全箇所で網羅する。条件分岐は排他的ではなく組み合わせで考える。仕様から削除した属性はUI→状態管理層→永続化→レンダリングの順に全て除去する。

#### 6.3.7 JavaScriptのTruthiness/Falsinessの罠

- 教訓: 空配列[]や空オブジェクト{}はtruthyである。||での短絡評価に頼らず、型や状態を明示的に判定する。TypeScriptの型システムを活用すればこの問題の大部分は防げる。

#### 6.3.8 イミュータブルエンティティのAPI不備

- 教訓: withXxx系メソッドを導入する際、全属性について必要なメソッドが網羅されているか洗い出す。呼び出し側と提供側の変更が揃っているかテストで検証する。

#### 6.3.9 イミュータブルオブジェクトと可変データの混在

- 教訓: 世界データの復元では、後続処理が期待するデータ形式（イミュータブル vs 可変）を把握し、適切な形で返す。Object.freeze済みのエンティティを共有キャッシュに直接入れて後続が書き込もうとすると例外になる。

#### 6.3.10 初期データのエンティティ化漏れ

- 教訓: ドメインエンティティ化したデータ構造に依存するユースケースを導入したら、初期データ・テストデータ・スタブも全て同じエンティティ型にする。デフォルト生成コードは見落としやすい。

### 6.4 共有データと永続化

#### 6.4.1 検証前の共有キャッシュ破壊

- 教訓: 結果が確定するまで共有キャッシュの世界データを破壊的に更新しない。検証用の仮データを作り、成功が確認できた段階でのみ反映する。永続化が失敗してもキャッシュが汚染されないようにする。

#### 6.4.2 頂点差し替え時のロールバック漏れ

- 教訓: 頂点IDを差し替える処理では、検証前にworld.featuresを書き換えない。失敗時は追加した頂点と地物の両方をロールバックする。

#### 6.4.3 共有頂点化の保持頂点選択

- 教訓: 共有頂点化で残す頂点をIDの古さだけで決めず、操作コンテキスト（ドラッグ対象かどうか）に基づいて決定する。

#### 6.4.4 リング階層の削除ルール

- 教訓: リング階層削除ではリング種別ごとの許容親を尊重する。親領土リング削除時は直下の穴をカスケード削除し、領土リングのみを親の親へ付け替える。

#### 6.4.5 ID生成の衝突

- 教訓: ミリ秒精度のタイムスタンプだけに依存したID生成は同一ミリ秒内で衝突する。ID生成サービスを経由し、既存データとの重複検査を行う。
