# 地物時空変遷編纂機 - 開発ガイド

## 重要な注意事項

> **フェーズ注記**: 以下の注意事項は主に保守・不具合修正フェーズを想定した制約である。初期実装フェーズでは「変更箇所を最小限にとどめる」「既存コメントの温存」等の制約は必要に応じて緩和される。ただし「根本原因の修正」「省略の禁止」「教訓の記録」は全フェーズで有効。

- 余計な変更は行わないこと。
- 指示されていない変更をしないこと。
- 変更箇所を最小限にとどめること。
- 既存のコメントは温存すること。
- コメントを記載するときは、本質的に必要な内容を書くこと。一時的な情報を含めないこと（「// 修正 :～」「//ここを修正」「//--修正ここから--」などの書き方をしないこと）。
- 論ずるときは、根拠となるソースコードを直接引用すること。
- コードの修正時は、「～も同様に～」といった省略は一切禁止。
- 不具合修正時は、必ず根本原因を修正すること。場当たり的な対応をしないこと。
- まずは原因の解析に注力すること。
- 実装上の失敗は開発ガイドに保存していくこと。
- 実装のやり残しは現状.mdに追記していくこと。
- **文書の役割分担**:
  - `現状.md` = 未解決課題・既知不具合・残作業・テスト総数。完了した課題は原則として即時削除すること。
  - `実装済み.md` = 実装履歴・フェーズ進捗・完了した機能の記録。
  - 例外として、同一課題を複数回の作業で分担し、削除すると進行文脈が失われる場合にのみ現状.mdに一時的に残してよい。行頭に[完了]とつけ、残作業完了時に課題全体を削除する。
  - 必要な履歴や再発防止策は本ガイドの「開発上の注意点とよくある落とし穴」に移すこと。
- 現状.mdから記載を削除する場合、それがリストの最後でない限り欠番として残すこと。
- 要件定義書は、仕様本文（完全日本語）と内部実装補遺を分離して記述すること。
  - 仕様本文では英語識別子・内部キー名・クラス名・コード値を直接記載しないこと。
  - 実装詳細は内部実装補遺に集約すること。

## 1. アーキテクチャ概要

本アプリケーションはドメイン駆動設計（DDD）の原則に基づいて構築される。

### 1.1 技術スタック

| レイヤー                   | 技術                         |
| -------------------------- | ---------------------------- |
| デスクトップフレームワーク | Electron                     |
| 言語                       | TypeScript (strict mode)     |
| UIフレームワーク           | Svelte 5 (Runes API)         |
| 状態機械                   | XState v5                    |
| レンダリング               | SVG                          |
| データ形式                 | JSON（ローカルファイル保存） |
| ビルド                     | Vite (electron-vite)         |
| テスト                     | Vitest + Playwright          |
| 幾何学計算                 | polygon-clipping             |

### 1.2 ドメイン層

アプリケーションの中核となるビジネスロジック。

- エンティティ: Vertex, Feature, Point, Line, Polygon, Layer
- 値オブジェクト: Coordinate, TimePoint, FeatureAnchor, PropertyProjection, Ring
- ドメインサービス: GeometryService, TimeService, LayerService
- 集約ルート: World

### 1.3 アプリケーション層

ユーザー操作をドメインロジックに変換し、結果を表示層に返す。

- ユースケース: EditFeatureUseCase (facade), AddFeatureUseCase, UpdateFeatureAnchorUseCase, DeleteFeatureUseCase, VertexEditUseCase, NavigateTimeUseCase, ManageLayersUseCase
- PolygonEditService: ポリゴンのリング構造編集
- PrepareFeatureAnchorEditUseCase, ResolveFeatureAnchorConflictsUseCase, CommitFeatureAnchorEditUseCase
- CQRSの原則: コマンド/クエリ/イベントの分離

### 1.4 インフラストラクチャ層

外部リソースへのアクセスやフレームワーク統合。

- JSONWorldRepository, FileSystem, JSONSerializer
- SVGRenderer (split into ViewportManager, FeatureRenderer, StyleResolver, LabelRenderer)
- Logger, ConfigManager

### 1.5 プレゼンテーション層

ユーザーインターフェースとユーザーとの対話。Svelteコンポーネントを使用。

- コンポーネント: MapCanvas, TimelinePanel, Toolbar, Sidebar, ConflictResolutionDialog
- 状態管理: XState v5 ステートマシン (toolMachine, conflictResolutionMachine, timelineMachine)
- XState v5 は `createActor` ベースで手動統合する（`@xstate/svelte` はSvelte 5未対応のため不採用）。ユーザー操作はイベントとしてマシンに送信する

## 2. 実装ガイドライン

### 2.1 一般的なコーディング規約

- TypeScript strict mode を有効にし、any を禁止
- インデントは2スペース
- セミコロン使用
- 命名規則:
  - クラス名: PascalCase
  - 変数/関数: camelCase
  - 定数: UPPER_SNAKE_CASE
  - プライベートメンバー: TypeScriptのprivate修飾子を使用（アンダースコア接頭辞ではなく）
- 関数は1つのことだけを行い、30行以下に抑える
- ネストを3レベル以下に抑える
- コメントは日本語で記述
- 1ファイルあたり300行を目安に分割（Svelteコンポーネントは200行以下、2.5節参照）

### 2.2 ドメイン層の実装ガイドライン

- 不変性: エンティティと値オブジェクトはreadonlyプロパティで設計。状態変更は新インスタンスを返す。
- 値オブジェクトの等価性: equals()メソッドによる属性値比較。
- ドメインサービスはステートレス。
- ドメインロジックは他の層に漏れ出さない。
- 集約ルートの整合性を保つ。集約間参照はIDで。

### 2.3 アプリケーション層の実装ガイドライン

- CQRS原則に従い、コマンドとクエリを分離。
- ユースケースは特定のタスクに集中。
- トランザクション管理（操作の原子性保証）。
- イベント駆動で状態変更を通知。

### 2.4 インフラストラクチャ層の実装ガイドライン

- 外部依存性はこの層に閉じ込める。
- アダプタパターンで抽象化。
- リポジトリは永続化詳細を隠蔽。
- 低レベル例外はアプリケーション固有の例外に変換。

### 2.5 プレゼンテーション層の実装ガイドライン

- Svelteコンポーネントは表示とイベント受付に専念。
- ビジネスロジックはコンポーネントに書かない。ユースケースに委譲。
- XState v5の状態機械でツール状態を管理。状態遷移を宣言的に定義し、不正な遷移を起こしにくくする。
- コンポーネントは200行以下を目安に分割。

### 2.6 テスト戦略

- ドメイン層: jsdom不要の純粋な単体テスト（Vitest）。GeometryServiceはカバレッジ100%目標。
- アプリケーション層: DIで依存を差し替えてテスト。
- UI: Playwright（E2E）で主要シナリオのみ。
- テストは tests/ 配下に、ソースと同じディレクトリ構造で配置（例: `tests/domain/entities/Feature.test.ts`）。
- **テストは実装と同時に書く。** 実装後にテストを後回しにしない。
- 現状.md の実装済みテーブルにはテスト有無カラムを維持し、件数を記録する。

### 2.7 ビルド・テスト環境の注意点

- `vitest.config.ts` にパスエイリアス（`@domain`, `@application` 等）を定義する必要がある。`electron.vite.config.ts` のエイリアスと二重管理になるため、片方を変更したらもう片方も追従すること。

## 5. 開発フロー

### 5.1 開発プロセス

要件分析→設計→実装（TDD）→テスト→デプロイ→ドキュメント改善

**開発フローに改善が見つかったら、即座にこのセクションに還元すること。** ワークフローの知見はコードと同じく資産であり、会話の中に埋もれさせない。

#### 5.1.1 1機能の作業フロー

1. 実装する機能を宣言し、要件定義書の該当セクションを引用する。
2. 本ガイド 5.1.2 の「AI作業開始ゲート」に従い、関連教訓を引用・適用宣言する。
3. 実装する。
4. テストを書き、全パスを確認する。
5. `npm run build` でビルドが通ることを確認する。
6. 現状.md を更新する（実装ステータス）。テスト件数は pre-commit フックが自動更新するため手動更新不要。
7. コミットする。

#### 5.1.2 AI作業開始ゲート

このリポジトリで作業するAIは、実装・修正・リファクタリング・設計提案に入る前に、必ず以下を行う。

1. `開発ガイド.md` §6 から今回の作業に関連する教訓を最低1件引用する。
2. 引用は節番号・見出し・教訓本文を含める。要約だけで済ませない。
3. 引用直後に、その教訓を今回どの判断に使うかを「適用:」として1文以上で明示する。
4. `実装済み.md` の近い履歴に原因が記録されているのに §6 に対応する教訓がない場合は、作業前または作業と同時に §6 へ教訓化する。
5. 完了報告では、実装済み.mdへの履歴追記だけでなく、§6へ新規教訓を追加すべき根本原因があったかを明示する。

推奨フォーマット:

```text
今回参照する教訓:
- 開発ガイド§6.x.y「見出し」: 「教訓: ...」
適用: 今回は ... なので、... を確認してから変更する。
```

## 6. 開発上の注意点とよくある落とし穴

ここからが最も重要なセクション。過去の開発で得られた教訓を記録。

### 6.0 教訓の使い方と更新ルール

- 教訓は「読んだ」だけでは作業開始条件を満たさない。作業前の応答で、関連する節番号・見出し・教訓本文を直接引用し、今回の実装判断へどう適用するかを明示する。
- `実装済み.md` は履歴であり、再発防止策の置き場ではない。不具合の根本原因・見落としやすい判断基準・回帰テスト観点は、本節へ移す。
- 既存教訓で説明できる不具合なら、完了報告で参照した教訓を明示する。既存教訓で説明できない不具合なら、新しい教訓を追加してから完了扱いにする。
- 教訓は具体的な禁止・確認観点として書く。「注意する」「気をつける」だけで終わらせず、どの状態・層・テストで検出するかまで記録する。
- 一時的な実装状況や進捗はここに書かない。進行中の課題は `現状.md`、完了履歴は `実装済み.md`、再発防止の恒久知識は本節に置く。

### 6.1 状態管理と更新ロジック

#### 6.1.1 状態間の依存関係の更新漏れ

- 教訓: ある状態が変化したら、それに依存する他の状態も検証・更新する。例えば「時間が変わったら、選択中の地物がその時間に存在するか検証し、存在しなければ選択解除」。暗黙的ルールはコードで明示的にチェックする。

#### 6.1.2 非表示要素の選択・操作の混入

- 教訓: ユーザー操作の対象は「現在UI上に表示されている要素」に限定する。データソース全体を直接検索せず、Svelteのリアクティブ状態（`$derived`等）が管理する表示状態を基準にする。

#### 6.1.3 Undo/Redoスタック操作の副作用

- 教訓: pushUndoがclearRedoStackを呼ぶような副作用を持つメソッドは、全呼び出しコンテキストで副作用が適切か慎重に検討する。Redo操作からのpushUndoではRedoスタックをクリアしてはならない。

#### 6.1.4 コンポーネント間での状態同期漏れ

- 教訓: 共有リソースを更新したら、依存する全コンポーネントに変更を通知する。イベントバスで汎用的な状態変更イベントを発行し、各コンポーネントがリフレッシュするパターンが有効。リアクティブ状態やXStateコンテキストは内部データが常に最新とは限らない前提で設計する。

#### 6.1.5 プロジェクト設定とアプリ設定の責務混在

- 教訓: プロジェクト固有設定とアプリ全体設定の境界を先に定義し、保存先と反映先を一貫させる。

#### 6.1.6 UI文脈を既定値で上書きする追加・編集処理

- 教訓: 追加・編集・複製・結合の対象は、現在のUI文脈（フォーカス中レイヤー、選択中地物、現在時刻、表示中wrapなど）を第一候補にする。コレクションの先頭要素や固定デフォルトを使うのは、UI文脈が存在しない場合だけに限定する。特にレイヤーフォーカス中の追加処理では、追加先レイヤーの解決をUI状態からテストし、先頭レイヤーへ落ちる回帰を防ぐ。

#### 6.1.7 時間操作の粒度と速度の二重適用

- 教訓: 時間軸UIでは、年・月・日を分解した一部の値だけを下位層へ渡さず、`TimePoint` のような完全な時間値を受け渡す。再生処理では、タイマー間隔と1tickあたりの進行量のどちらが速度を表すのかを一箇所に決め、両方へ倍率を掛けない。月日ステップ、再生速度、プロジェクト設定の時刻範囲は同じヘルパーで検証し、10x再生などの回帰テストを置く。

### 6.2 UIインタラクションとユーザー操作フロー

#### 6.2.1 操作フローの完了トリガー欠落

- 教訓: UIインタラクションのフロー（開始・途中操作・完了/キャンセル）では、必ず「完了」のトリガーを定義・実装する。ユースケースにメソッドがあってもUIからの呼び出しがなければ機能しない。

#### 6.2.2 排他的イベント処理による意図しない動作

- 教訓: 排他的な条件分岐が他の重要な処理をスキップさせていないか確認する。頂点クリック時にその地物の選択も行うべきか等、選択優先順位と同時/排他選択を明確に定義する。

#### 6.2.3 オーバーレイ上のモーダルUIでのイベント競合

- 教訓: 地図の上にモーダルを表示する場合、イベント発生源を検証し、モーダル内のイベントが地図操作をトリガーしないようにする。stopPropagationだけでは不十分な場合があり、リスナー内でevent.targetを判定する。

#### 6.2.4 選択ハイライトで穴が塗り潰される

- 教訓: 穴や飛び地を持つポリゴンの塗りでは、領土リングと穴リングを同じpathのサブパスとして定義し、fill-rule="evenodd"を適用する。本体描画とオーバーレイ描画でリング階層の解釈を共有する。

#### 6.2.5 選択状態の分類漏れ

- 教訓: 複数の選択状態（単一・複数・不明・空）を扱うUIでは、派生済みのヘルパー結果だけでなく、生のIDセットを全て評価して分岐を網羅する。仕様のメッセージとViewの条件分岐を一対一で対応付ける。

#### 6.2.6 設定フォームのスクロール領域不足

- 教訓: フォームの操作ボタンが下部に配置されるUIでは、親の高さとoverflowを明確にし、min-height: 0とoverflow-y: autoを設定する。最小ウィンドウサイズでボタンが見切れていないか確認する。

#### 6.2.7 マウスイベントシーケンスの状態管理不備

- 教訓: mousedown→mousemove→mouseupのシーケンスで、全てのパス（クリック、ドラッグ）で状態が正しく開始・更新・終了されることを保証する。mousedownで設定した状態はmouseupで必ず解除する。mousemoveが発生しなかった場合（クリック）の考慮漏れに注意。

#### 6.2.8 一時的なインタラクション状態と描画の連携不備

- 教訓: 複数ステップのUIインタラクションでは、一時的な状態をXStateマシンのコンテキストで一元管理し、プレビュー描画はそのコンテキストから状態を取得して反映する。
- 教訓: XState外に補助的な一時状態を置く場合は、描画座標と同じ開始・確定・キャンセル・ツール切替・モード切替の全経路で明示的に破棄する。HUDの選択値やドラフト設定が残ると、次回の操作へ暗黙に引き継がれ、ユーザーが意図しない親指定や設定で確定できてしまう。E2Eでは「選択→キャンセル→再作成」と「選択→確定→再作成」の連続操作を確認する。
- 教訓: Svelteの`<select>`はoption DOMの再利用で現在値が残ることがある。補助状態を減らす場合でも、描画セッション単位のkey化と`{#each ... (id)}`でoptionを安定識別し、E2Eで「propは既定値に戻ったがselectの表示だけ残る」ケースを確認する。

#### 6.2.9 条件付き描画内の未定義参照による更新停止

- 教訓: 選択時やモード切替時にだけ描画されるUI分岐では、渡すprop名とローカル変数名の不一致が `ReferenceError` を起こし、無関係なリアクティブ更新停止に見えることがある。条件付きで出現するツールバーやオーバーレイは、表示契機を含むE2Eで実際に描画させ、ブラウザ実行時エラーも検出する。
- 診断ポイント: 「値は変わっているのに DOM が反映されない」症状が出たら、まずブラウザコンソールの `ReferenceError`/`TypeError` を確認する。`$.set()` やストアの値が更新されていても、フラッシュ中の未捕捉エラーで後続の `$effect` や DOM 更新が全停止する。`console.log` では値が読めるため「リアクティビティ故障」と誤診しやすい。E2Eでは `page.on('pageerror')` を設定してフラッシュ中のエラーを捕捉する。

#### 6.2.10 横方向無限スクロールでオーバーレイだけが追随しない

- 教訓: 横方向無限スクロールでは、ベース地図や地物本体だけでなく、頂点ハンドル・描画プレビュー・測量線のようなSVGオーバーレイも同じ `wrapOffset` 描画文脈に置く。地図本体とオーバーレイでラップ位置が分離すると、複製地図上に操作対象が出ず、パン後に「表示されない」「ずれて見える」不具合になる。

#### 6.2.11 東西端またぎの描画と無限スクロール本体は別々に担保する

- 教訓: 頂点や測量点の保存座標は `-180〜180` に正規化してよいが、`LineString` / `Polygon` / 大円パスの描画・ヒットテストでは、点列を前後関係からアンラップして `179° → -179°` を短い経路として扱う。同時に `wrapOffset` は `-360 / 0 / +360` の固定列挙ではなく、現在の `viewBox` と交差する全タイルを動的に生成する。保存用正規化座標をそのままパス化したり、隣1枚だけを描画すると、東西端またぎが世界横断線になり、1周以上のパンで空白が出る。

#### 6.2.12 巻き戻し前提のプレビューで正規化・クランプをしない

- 教訓: ドラッグ中プレビューのように「中間状態を一時適用し、確定前に逆変換で巻き戻す」処理は、可逆でなければならない。`normalize()` や緯度クランプをプレビュー段階へ入れると、東西端またぎで頂点ごとに別周回へ飛んで描画経路が壊れたり、`y > 90` のような一時値が失われて逆移動で元に戻らなくなる。正規化・クランプ・保存形式への整形は、コミット処理や永続化直前の責務として分離する。

#### 6.2.13 非正規化プレビュー座標は描画時だけ主表示帯へ戻す

- 教訓: 6.2.12 の都合でプレビュー中の経度が `±180` を超えること自体は許容されるが、そのまま `geoToSvgX()` へ流すと、図形やハンドルが別タイルの overflow に依存して表示され、`wrapOffsets` の境界でジャンプする。`LineString` / `Polygon` はアンラップ後の経度列全体を、`Point` やハンドルは単点経度を、SVG化直前に主表示帯（`-180〜180`）へ寄せてから描画する。プレビュー状態の保存値を変更してはいけない。

#### 6.2.14 タイル単位クリップは局所座標化なしで導入しない

- 教訓: `wrapOffset` ごとの `<g transform="translate(..., 0)">` の内側に `overflow="hidden"` なネスト SVG を入れるだけでは、地物がまだワールド全体の SVG 座標で描かれている限り、180°子午線またぎの面や線をタイル境界で切断してしまう。タイル単位クリップを使うなら、そのレイヤー全体を各タイルの局所座標系へ射影してから描画すること。そこまで行わない場合は、背景・ベースマップ・地物・グリッド・各種オーバーレイを「レイヤー単位で全タイル分まとめて描く」順序に再構成し、隣接タイルの背景やグリッドが、はみ出して描かれる地物を後から上書きしないようにする。

#### 6.2.15 頂点ドラッグ開始座標に保存経度をそのまま使わない

- 教訓: 180°超の経度幅を持つ面や共有頂点では、保存中の頂点経度と、ユーザーが実際に掴んだハンドルの表示経度が別周回になることがある。頂点ドラッグ開始座標やエッジ中点は、`VertexHandles` が描画に使ったアンラップ済み経度をそのままコールバックへ渡し、ドラッグ中もその周回に追従して更新すること。頂点移動・共有頂点移動で経度を `normalize()` すると、この周回情報が失われて接続方向が反転しやすい。経度正規化は、地物全体の平行移動や永続化直前など、周回情報を保持しなくてよい責務に限定する。

#### 6.2.16 経度は生値で保存し、描画時にunwrapしない（経度生値設計）

- 教訓: 経度は数直線として扱い、円環性は表示層だけが知る。
    - **保存**: 頂点の経度は生値のまま保存する。`190°` は「180°から東に10°」を意味する。`normalize()` は保存・移動・描画の経路で呼ばない。
    - **描画**: `buildRingPath` / `buildPolygonPath` / `buildLinePoints` で `unwrapLongitudeSequence` を使わない。頂点の生値座標に同一のタイルオフセットを加算して `geoToSvgX()` するだけにする。頂点間の相対位置が方向情報を自然に持つため、短い弧の仮定が不要になる。
    - **ラベル重心**: `labelRendererUtils` でも `unwrapLongitudeSequence` を外し、生値座標の算術平均を取る。結果を表示帯へ折り返す。
    - **頂点追加・クリック**: SVG→geo変換で既存リングの座標系と整合する経度を返す。`wrapOffset` 付きタイル上のクリックはそのタイル基準の生値になる。
    - **正規化の使用箇所**: ステータスバーの座標表示、手入力値の変換など、ユーザー向け表示に限定する。
    - **2周超・排他判定**: 720°超の地物や描画中プレビューは、固定枚数のwrap描画ではなく、地物の生値経度範囲と現在viewBoxに基づいて必要なタイルを動的算出する。排他判定・差分適用も `-180〜180` 前提に戻さず、360°シフトを含めた候補範囲で評価する。
    - 参照: `../time-map-app-2` は同じ設計で動作しており、星をぐるぐる回るポリゴンの作成・編集が正常に機能する。

#### 6.2.17 表示専用モードに編集操作を残さない

- 教訓: 表示モードでは、編集HUD・頂点ハンドル・コンテキストメニューの編集項目・ショートカット（例: Ctrl+Aの頂点選択）を同じ権限判定で無効化する。UI部品ごとに個別判定を書くと、見た目だけ非表示でショートカットや右クリック操作が残る。モード境界は共通ヘルパーまたは状態機械で表現し、表示モードのE2Eでは「編集用DOMがない」「編集ショートカットが効かない」「選択表示は残る」を同時に確認する。

#### 6.2.18 スクリーン固定UIを世界座標サイズで描かない

- 教訓: 頂点ハンドル、エッジハンドル、スナップマーカー、測量マーカーのような操作UIは画面ピクセル基準で大きさを決める。SVGの度単位やワールド座標半径をそのまま使うと、ウィンドウ幅・ズーム・viewBoxの変化で巨大化または極小化する。描画直前に現在のviewBoxと画面サイズからピクセル→SVG単位へ変換し、ズーム/リサイズの回帰テストを置く。

#### 6.2.19 変換後の選択状態を一律クリアしない

- 教訓: 結合・分割・移動・削除のように対象IDや形状が変わる操作では、完了後の選択状態を固定の `clearSelection()` で決めない。コマンド結果が「残る地物ID」「新規地物ID」「削除されたID」を返し、UIはその結果から選択を再解決する。成功後もユーザーが次に調整すべき対象が残る操作では、選択維持を回帰テストに含める。

#### 6.2.20 外部SVGの座標系を固定スケール前提で読まない

- 教訓: ベースマップやユーザー指定SVGは、特定ファイルの座標スケールに依存せず `viewBox` / `width` / `height` から正規化する。固定の出界図スケールや暗黙サイズに合わせて読み込むと、別SVGへの差し替え時に地図位置・操作座標・ラップ描画がずれる。不適合SVGのフォールバックと、正規化後の 360x180 表示をテストする。

#### 6.2.21 拒否理由・警告メッセージの寿命を無関係操作で消さない

- 教訓: 検証拒否や保存失敗など、ユーザーが次の判断に使うメッセージは、発生元と関係のないクリックやマウス操作で即時消去しない。メッセージを消す契機は「同種操作の再試行」「明示的な閉じる」「成功した置換操作」のように意味を持つイベントへ限定する。特に地図クリックの共通ハンドラで一律に通知状態を初期化しない。

#### 6.2.22 視覚レイヤーと入力レイヤーの重なりを同時に設計する

- 教訓: 測量マーカー、頂点ハンドル、メニューのバックドロップ、モーダルのような重なり合うUIは、描画順だけでなく入力を受ける層も同時に決める。透明バックドロップや測量オーバーレイが編集ハンドルを覆うと、見た目は正しくても操作が奪われる。E2Eでは表示確認に加えて、重なった状態で背面/前面の期待操作が通ることを確認する。

#### 6.2.23 SVG上の巨大表示を実マーカーだけと決めつけない

- 教訓: SVG上で大きな丸・枠・ハイライトが見える場合、アプリが描いた `circle` / `rect` / `path` のサイズだけでなく、ブラウザ既定のフォーカスアウトライン、選択ハイライト、CSS疑似状態、ドラッグ中クラスを最初に切り分ける。ズームしても画面上の大きさが変わらない視覚物は、SVGの世界座標サイズではなくCSS/UA由来の可能性が高い。広範囲のマーカーサイズ変更に入る前に、スクリーンショット、DOMの `boundingBox`、`getComputedStyle()`、該当状態のE2Eで「実要素の半径」と「周辺ハイライト」のどちらが巨大なのかを固定する。

#### 6.2.24 Svelte 5 ルーン値の参照と関数呼び出しを混同しない

- 教訓: `$derived(expr)` と `$derived.by(callback)` はどちらも値を返すルーンであり、結果は値として参照する。`$derived(() => fn)` のように右辺が関数式であれば結果は「関数」であり呼び出し側で `fn()` と書くが、`$derived.by(() => value)` の結果は callback の評価結果（値そのもの）であり `value()` のように呼び出してはいけない。同じ画面でも普段通る経路では問題が出ず、ある条件分岐（例: `placement.childIds.length > 0` のときだけ走る派生形状計算）に入ったときだけ `TypeError: $.get(...) is not a function` を投げ、§6.2.9 の通りフラッシュ中の未捕捉エラーで以降の DOM 反映が停止する。
- 検出観点: 条件付き描画やサイドパスを含むコンポーネントでは、その条件を実際に通す E2E を用意し、`page.on('pageerror')` でランタイムエラーを必ず捕捉する。`afterEach` で `pageErrors` が空であることを断言するパターンを使う。型チェックは通ってしまうため、テスト側で押さえる必要がある。

### 6.3 データ整合性と型

#### 6.3.1 ドメインオブジェクトの代わりにリテラルオブジェクトを使用

- 教訓: DDDでは、エンティティや値オブジェクトは定義されたクラスのインスタンスとして扱う。プレーンオブジェクトを渡さない。TypeScriptの型システムでこれを強制する。

#### 6.3.2 コンポーネント間のデータ構造不一致

- 教訓: コンポーネント間でデータを渡す際、プロパティ名・型・構造が受け取り側の期待と一致しているか確認する。TypeScriptのインターフェースで契約を明示する。

#### 6.3.3 コンテキスト上の時刻とシステム時刻の混同

- 教訓: 時間軸を持つアプリでは、データのタイムスタンプは必ずコンテキスト上の時刻（タイムラインの現在時刻）を参照する。時刻源を一元管理するサービスから取得する。

#### 6.3.4 継承関係におけるメソッドオーバーライド漏れ

- 教訓: クラス継承でサブクラスが追加状態を持つ場合、基底クラスのwithXxx系メソッドはサブクラスでオーバーライドが必要か慎重に検討する。

#### 6.3.5 Undo/Redoのデータ復元

- 教訓:
  - 「操作を元に戻す」に必要な情報を正確に特定し、依存データも含めて過不足なく保存する。
  - 履歴にはプレーンオブジェクトを保存し、復元時は必ず新しいドメインインスタンスを生成する。
  - デシリアライズは再帰的に行い、ネストされた値オブジェクトも正しく変換する。
  - 履歴データのフォーマットは一貫させる。

#### 6.3.6 データ構造と描画ロジックの不整合

- 教訓: データ構造の全有効状態組み合わせを、描画・シリアライズ・操作の全箇所で網羅する。条件分岐は排他的ではなく組み合わせで考える。仕様から削除した属性はUI→状態管理層→永続化→レンダリングの順に全て除去する。

#### 6.3.7 JavaScriptのTruthiness/Falsinessの罠

- 教訓: 空配列[]や空オブジェクト{}はtruthyである。||での短絡評価に頼らず、型や状態を明示的に判定する。TypeScriptの型システムを活用すればこの問題の大部分は防げる。

#### 6.3.8 イミュータブルエンティティのAPI不備

- 教訓: withXxx系メソッドを導入する際、全属性について必要なメソッドが網羅されているか洗い出す。呼び出し側と提供側の変更が揃っているかテストで検証する。

#### 6.3.9 イミュータブルオブジェクトと可変データの混在

- 教訓: 世界データの復元では、後続処理が期待するデータ形式（イミュータブル vs 可変）を把握し、適切な形で返す。Object.freeze済みのエンティティを共有キャッシュに直接入れて後続が書き込もうとすると例外になる。

#### 6.3.10 初期データのエンティティ化漏れ

- 教訓: ドメインエンティティ化したデータ構造に依存するユースケースを導入したら、初期データ・テストデータ・スタブも全て同じエンティティ型にする。デフォルト生成コードは見落としやすい。

#### 6.3.11 preload/main/renderer のAPI契約不一致

- 教訓: renderer から呼ぶ `window.api`、preload が公開する関数名、main のIPCチャンネル名は、文字列の偶然一致に任せない。共有型・型付きラッパー・preload/main の単体テストで契約を固定する。API名変更時は renderer / preload / main / テストを同じ変更単位で更新し、旧名（例: `window.fileAPI`）が残っていないか検索する。

#### 6.3.12 列挙値・識別子の表記ゆれ

- 教訓: 地物種別、リング種別、モード名、設定キーのような分岐識別子は、表示文字列や思い込みで判定せず、型定義・定数・ファクトリを経由する。`Line` と `line` のような大小文字差でもUIバッジ、描画確定可否、保存形式が分岐する。新しい識別子を追加・改名したら、作成、表示、編集、保存、検索の全経路で同じ値を使うテストを置く。

### 6.4 共有データと永続化

#### 6.4.1 検証前の共有キャッシュ破壊

- 教訓: 結果が確定するまで共有キャッシュの世界データを破壊的に更新しない。検証用の仮データを作り、成功が確認できた段階でのみ反映する。永続化が失敗してもキャッシュが汚染されないようにする。

#### 6.4.2 頂点差し替え時のロールバック漏れ

- 教訓: 頂点IDを差し替える処理では、検証前にworld.featuresを書き換えない。失敗時は追加した頂点と地物の両方をロールバックする。

#### 6.4.3 共有頂点化の保持頂点選択

- 教訓: 共有頂点化で残す頂点をIDの古さだけで決めず、操作コンテキスト（ドラッグ対象かどうか）に基づいて決定する。

#### 6.4.4 リング階層の削除ルール

- 教訓: リング階層削除ではリング種別ごとの許容親を尊重する。親領土リング削除時は直下の穴をカスケード削除し、領土リングのみを親の親へ付け替える。

#### 6.4.5 ID生成の衝突

- 教訓: ミリ秒精度のタイムスタンプだけに依存したID生成は同一ミリ秒内で衝突する。ID生成サービスを経由し、既存データとの重複検査を行う。

#### 6.4.6 共有頂点化の参加制限単位

- 教訓: 共有頂点化の禁止条件を「地物単位」に広げない。要件がリング/線形単位で定義されている場合、ドメイン判定・UI候補探索・回帰テストのすべてを同じ粒度で揃える。同一地物でも別リングなら許可し、同一リング/同一線形だけを拒否する。

#### 6.4.7 新規プロジェクト時の保存コンテキスト引き継ぎ

- 教訓: 新規プロジェクト開始時は、表示中データだけでなく保存ユースケースが保持するファイルパス・メタデータ・バックアップ対象も同時に初期化する。UI state だけを空にすると、次回 `保存` が旧ファイルを上書きしたり、旧プロジェクト設定を持ち越す。

#### 6.4.8 自動バックアップ仕様の層間不一致

- 教訓: 自動バックアップの仕様が「保存場所・世代管理・命名規則」を含む場合、renderer のユーティリティだけで完結させず、preload/main のファイルI/O契約と一緒に更新する。固定連番ローテーションをローカル実装すると、`/savebackup/` 配下の実ファイル一覧と乖離しやすい。バックアップ作成時は、実ディレクトリ上のタイムスタンプ付きファイル一覧を基準に保持上限を判定する。

#### 6.4.9 共有頂点付き移動のトランザクション境界

- 教訓: 地物移動・複数頂点移動・共有頂点ドラッグでは、頂点座標だけでなく共有頂点グループの `vertexIds` と `representativeCoordinate` も同じトランザクションで更新する。Undo/Redoコマンドは変更前後の共有グループ状態を保存し、存在しない対象へのno-op実行で共有グループを消さない。プレビュー中は開始時スナップショットから復元し、クランプや制約処理を通っても元座標へ戻れることをテストする。

#### 6.4.10 未保存変更とファイル文脈をUI stateだけで管理しない

- 教訓: 保存済みパス、プロジェクトメタデータ、バックアップ対象、dirty state、close guard は同じ保存文脈として扱う。画面上のWorldだけを初期化・更新しても、保存ユースケースやmain側の終了確認が古い状態を持っていれば旧ファイル上書きや未保存警告漏れが起きる。新規作成、開く、保存、閉じるの各経路で renderer と main/preload の状態同期を確認する。

#### 6.4.11 頂点削除時の共有グループ整理

- 教訓: 頂点削除では形状からIDを外すだけでなく、削除後にどの地物からも参照されなくなった頂点を `vertices` から削除し、`SharedVertexGroup.vertexIds` からも同じトランザクションで除去する。グループが1頂点以下になった場合は共有関係として残さない。Undo/Redoコマンドは削除前の地物・頂点・共有グループ状態を一括で保存し、共有頂点を含む頂点削除の回帰テストを置く。

#### 6.4.12 Redoで生成IDを採番し直さない

- 教訓: Undo/Redo対象コマンドが頂点ID・リングID・地物IDなどを生成する場合、redoでUseCaseを再実行して新IDを採番し直してはいけない。初回executeで生成されたIDと変更後スナップショットを保持し、redoでは同じIDを復元する。後続コマンドがそのIDをparamsに保持する操作列（挿入→削除→Undo×N→Redo×N）を回帰テストに含める。

#### 6.4.13 共有辺を切る分割の隣接地物更新

- 教訓: ナイフ分割などで共有辺の途中に新しい切断点が生じる場合、分割対象地物だけに頂点を作って共有グループを更新しても不十分である。共有先地物の該当辺にも同座標の頂点を挿入し、その頂点を分割対象側の新頂点と同じ `SharedVertexGroup` に入れる。ただし、共有先へ挿入する座標は分割結果に含まれる全境界座標ではなく、分割片A/Bの両方に現れる切断境界座標へ絞る。共有先地物だけが共有端点間に中間頂点を持つ場合も、共有端点同士を直接セグメント判定するのではなく、共有端点間の直線チェーン上の各セグメントを探索する。既存頂点が切断点そのものにある場合は、新規頂点を作らずその既存頂点を共有グループへ入れる。その既存頂点が別グループに所属済みなら、1頂点1グループ制約を壊さないよう新規グループではなく既存グループへ分割側頂点を統合する。Undo/Redoでは、分割対象・新地物・共有先地物・追加頂点・共有グループを同じスナップショットとして復元する。回帰テストでは、共有辺の端点ではなく辺途中の切断点に共有先頂点が追加されること、同じ共有辺に複数の切断点がある場合に辺上の順序で挿入されること、共有辺上の既存中間頂点が非切断時に共有先へ挿入されないこと、共有先側だけに中間頂点がある場合も切断点が挿入されること、切断点と同座標の既存共有先頂点が再利用されること、既存共有先頂点の所属グループへ分割側頂点が統合されることを確認する。

### 6.5 テスト環境の落とし穴

#### 6.5.1 Playwright × Electron 41 の直接起動不能

- 教訓: Playwright 1.58 の `_electron.launch()` は Electron 41 (Chromium 136+) で動作しない。根本原因は二つあり、(1) Chromium 136 以降は `--remote-debugging-port=0` を CLI 引数として拒否するため Playwright が追加する引数で即終了する、(2) `-r loader.js` プリロードの `require('electron')` が Electron ビルトインではなく npm パッケージ側の CLI ラッパー（パス文字列を返す）に解決され `app` が undefined になる。`node_modules/electron/` の存在自体が原因で、Electron ダウングレードやローダー差し替えでは回避できない。
- 回避策: dev サーバー経由のブラウザ E2E を採用する。`playwright.config.ts` で `reuseExistingServer: true` を設定し、ユーザーが VSCode ターミナルから `npm run dev` を起動した状態で `npm run test:e2e` を実行する（Git Bash から `npm run dev` を起動すると同じ `require('electron')` 問題で失敗するため、dev サーバー起動元は PowerShell/cmd に固定する）。
- 失うテスト範囲: Electron IPC（保存/読み込みダイアログ）、ネイティブメニュー（Menu API）、マルチウィンドウ、`app.getPath()` 等の Electron 固有 API。
- 復帰条件: Playwright PR #39012（`fix(electron): pass port via switches not args`）がリリースされ、かつ `require('electron')` 解決問題も解消されたバージョンが出たら `_electron.launch()` に戻す。それまで `scripts/patch-playwright-electron.js` は参考用に保持する。

#### 6.5.2 設定ファイルのモジュール形式を実行環境と揃える

- 教訓: Vite/Svelte/Playwright などの設定ファイルは、拡張子と `package.json` のモジュール種別を実行環境に合わせる。ESMとして読まれる設定を `.js` のまま置くと、ローカル環境やNodeバージョン差で `npm run check` が落ちる。設定ファイルのリネームや形式変更は、`npm run check` と関連ツールの起動確認まで同じ変更単位に含める。

#### 6.5.3 ツールが直接使う依存は transitive dependency に頼らない

- 教訓: テスト・型チェック・ビルド・スクリプトが直接 import / execute するパッケージは、別パッケージ経由の推移依存に頼らず `package.json` へ直接依存として宣言する。ツールチェーン更新時に推移依存の解決が変わると、コード変更と無関係に `npm run check` やテストが失敗する。

#### 6.5.4 E2Eで失った範囲は単体テストで補う

- 教訓: Electron直接起動E2Eを一時的に使えない間も、IPC、preload、ファイルI/O、close guard、未処理Promise rejectionのようなElectron固有経路を未検証にしない。main/preloadの純粋関数化・アダプタ分離・モックIPCでVitest対象にし、E2Eで見られない保存/読込失敗や終了確認を単体テストで固定する。

### 6.6 幾何制約・リング構造・編集コマンド

#### 6.6.1 検証をUIだけ・ドメインだけに偏らせない

- 教訓: 自己交差、同一レイヤー重複、リング階層破綻、共有頂点禁止条件のような編集制約は、UIの候補表示だけでなくコマンド・ユースケースの確定前にも同じサービスで検証する。UIが候補を隠しても、ショートカット、コンテキストメニュー、Undo/Redo、テスト用直接呼び出しで不正状態が入る。拒否理由はユーザー通知とテスト可能な戻り値の両方で扱う。

#### 6.6.2 穴・飛び地・複数領土リングを平坦化しない

- 教訓: ポリゴンのリング構造では、`territory` と `hole` の親子関係を常に保つ。領土リングが複数ある地物を再マージ・再分割するとき、2つ目以降を機械的にhole扱いしない。穴内の島はhole配下の `territory` として再構築し、占有面の判定では「領土リング + 直下の穴リング」を単位にする。描画、ヒットテスト、競合検出、結合、分割コマンド、ナイフ確定可否UI、保存のすべてで同じリング階層を使う。複数territoryの開線バリデーションでは交差回数を合算せず、少なくとも1つのterritory外周が2回以上交差することを条件にする。開線が横断していないterritoryは、延長半平面で分断せず、形状を丸ごと維持したまま所属先だけを判定する。

#### 6.6.3 結合対象の事前条件を明示する

- 教訓: 結合・割譲・合邦の対象は、同一地物ではないこと、同一レイヤーに属すること、面地物であること、時間上有効であることを操作開始前と確定前の両方で検証する。不正対象をモーダルやサービスへ流してから失敗させると、選択解除や部分更新の副作用が残りやすい。結合後は残る地物ID・追加リング・削除対象を結果として返し、UI選択とUndo情報をそこから再構築する。

#### 6.6.4 EdgeSlideは終点だけでなく移動線分全体を判定する

- 教訓: エッジ滑りや頂点ドラッグの制約では、移動先点が合法かだけでなく、開始点から移動先までの線分が障害物や境界を横切るかを判定する。障害物を横切る移動は通過させず、内部へ入る移動は最初に交差した境界を衝突応答の基準にする。境界上の始点から内部へ向かう移動を反対側エッジへ飛ばさず、移動先が境界上の合法接触であるケースは許可する。

#### 6.6.5 ブーリアン演算結果の複数片を落とさない

- 教訓: `polygon-clipping` などのブーリアン演算は、単一ポリゴンではなく複数ポリゴン・複数リングを返す前提で扱う。差分・結合・競合解決では、返却された全ての片を頂点・リング・親子関係へ変換し、Commit側で作成頂点とリングをまとめて反映する。Resolve側で共有キャッシュを直接書き換えず、`createdVertices` などの結果データとして返す。

#### 6.6.6 隣接・自動配色はレイヤーと表示経度の文脈を揃える

- 教訓: 隣接辺ベースの自動配色や境界共有判定は、全地物を一括で見ず、同一レイヤー・現在の表示経度文脈・生値経度設計に揃えて評価する。保存済み色と自動解決色の優先順位を明示し、180°子午線またぎの共有辺も同じ正規化/生値方針で比較する。

#### 6.6.7 衝突検出と滑り応答を分ける

- 教訓: EdgeSlideのようなドラッグ制約では、Time of Impactで最初の接触点を求める処理と、接触後の残り移動を処理する衝突応答を分ける。接触点で毎回止めると、辺上や頂点近傍で固着しやすい。頂点-辺は接触辺の接線へ、移動辺-障害物点は固定点-障害物点の接触線へ残り移動を射影し、その結果を再度制約判定する。`t=0` の既存接触は新規衝突として止め直さず、法線方向の食い込みだけを捨てて接線方向の移動を許可する単体テストを置く。ただし、辺-点滑りの候補線分が障害物面の内部を通る場合は、外周線上の点衝突として採用しない。障害物は外周線だけでなく面として再評価し、単に直前座標へ戻すのではなく、ドラッグ線分に近く、かつ移動辺が面内部を通らない境界候補へ退避させる。高速ドラッグでは目的地に近い候補を優先すると奥の角へ飛ぶか固着しやすいため、ドラッグ線分からの距離と進行方向で候補を選ぶテストを置く。
