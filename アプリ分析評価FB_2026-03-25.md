# アプリ分析・評価フィードバック

作成日: 2026-03-25
対象: `time-map-app-3`

## 1. 先に結論

このアプリは、ドメイン設計とテスト密度がかなり強いです。単なる雛形ではなく、歴史地図エディタとして十分に「使える核」を持っています。

一方で、評価を厳密にすると「強いベータ、まだ完成扱いには早い」です。理由は、以下の3系統の問題が残っているからです。

1. 破壊的操作が正規のユースケース経路を通らず、整合性・Undo・未保存管理を崩している。
2. プロジェクト設定が UI 上は存在するが、保存・読込・実際の表示／計算へ接続されていない。
3. Electron 固有のファイル操作系がブラウザ E2E の検証外にあり、その結果として自動バックアップ破損のような統合不具合が残っている。

総評としては以下です。

| 観点 | 評価 | コメント |
| --- | --- | --- |
| ドメイン設計 | 高い | 値オブジェクト、サービス、ユースケースの切り方は明確 |
| テスト品質 | 高い | 単体 1036 件、E2E 134 合格はかなり厚い |
| UI/機能量 | 高い | 地図表示、追加、編集、測量、タイムライン、設定まで一通り揃う |
| 統合品質 | 中 | 保存・設定・削除・バックアップの接続に穴がある |
| 保守性 | 中以下 | `App.svelte` を中心に責務集中が進みすぎている |
| 出荷準備度 | 中以下 | 主要機能は見えるが、データ保全系を直すまで安心して出せない |

## 2. 今回の確認内容

実施した確認は以下。

- `npm test`
  - 結果: 65 ファイル、1036 テスト通過
- `npm run build`
  - 結果: 通過
- `npm run test:e2e`
  - 結果: 139 テスト中 134 合格、5 fixme/skip
- `npm run check`
  - 表示上は `0 errors / 0 warnings`
  - ただし `svelte.config.js` 読み込み時に ESM/CJS 由来のノイズあり

また、`現状.md` でも以下の状態が明示されています。

```md
全1036ユニットテスト＋139 E2Eテスト（134合格＋5 fixme）。
```

```md
Electron直接起動は ... require('electron') 解決問題のため断念。
```

このため、現状の E2E は「ブラウザ実行での UI 検証」は強い一方、「Electron 固有の IPC / preload / ファイルアクセス」の実検証は弱い、という前提で評価しています。

## 3. 良い点

### 3.1 ドメインとアプリケーションの骨格はかなり良い

リポジトリ構成を見る限り、少なくとも意図としては DDD 4 層構造を真面目に守ろうとしています。`domain` / `application` / `infrastructure` / `presentation` が明確で、しかもそれぞれにテストがあります。

特に強いのは以下です。

- `FeatureAnchor`, `Ring`, `Layer`, `World` など、ドメインの中核概念が明示的
- `GeometryService`, `LayerService`, `SharedVertexService`, `KnifeService`, `MergeService` など、複雑な計算責務を UI から切り離せている
- `UndoRedoManager` とコマンド群があり、少なくとも主要編集操作の一部は「巻き戻せる形」で実装されている

これは「あとから直せる構造」が既にある、という意味で大きいです。

### 3.2 テスト密度は明確な強み

このリポジトリの最大の長所は、見た目ではなく検証密度です。

- 単体テスト 1036 件
- E2E 134 合格
- しかもドメイン、アプリケーション、インフラ、プレゼンテーション状態機械まで横断している

雑に UI だけ先に作ったプロトタイプではなく、ロジックの地盤をかなり固めています。

### 3.3 機能の広がりがある

単に「点・線・面を置ける」だけではなく、以下が揃っています。

- 横方向無限スクロール地図
- 追加／編集／測量モード
- 頂点操作、穴・飛び地追加、分割、結合
- レイヤー、プロパティ、地物一覧
- タイムライン再生
- プロジェクト設定
- 自動バックアップ、未保存警告、D&D 読込

この機能の広さは評価してよいです。

## 4. 重要な問題点

以下は重要度順です。

### 4.1 地物削除が `DeleteFeatureUseCase` を通っていない [重大]

#### 現状コード

`src/renderer/App.svelte`

```ts
onDelete: () => {
  // 地物削除（簡易）
  if (selectedFeatureId) {
    (addFeature.getFeaturesMap() as Map<string, Feature>).delete(selectedFeatureId);
    selectedFeatureId = null;
    selectedVertexIds = new Set();
    refreshFeatureData();
  }
},
```

#### 本来の設計側

`src/renderer/application/DeleteFeatureUseCase.ts`

```ts
// 地物を削除
features.delete(featureId);
deletedFeatureIds.push(featureId);
eventBus.emit('feature:removed', { featureId });

// 頂点クリーンアップ: 他の地物で使われていない頂点を削除
const usedVertexIds = this.collectAllUsedVertexIds(features);
const vertices = this.featureUseCase.getVertices() as Map<string, Vertex>;
const sharedGroups = this.featureUseCase.getSharedVertexGroups() as Map<string, SharedVertexGroup>;
```

#### 問題

右クリック削除が正規ユースケースを迂回しています。

その結果、少なくとも以下が抜けます。

- 未使用頂点のクリーンアップ
- 共有頂点グループの更新
- `feature:removed` 発火
- `UndoRedoManager` 経由の履歴化
- Dirty 状態との連動

#### 影響

- 画面上は消えても内部に孤立データが残る可能性がある
- 保存データの整合性に悪影響を与える可能性がある
- 削除操作だけ Undo できないという UX 不整合が起こる
- 以後の編集ロジックが「消えたはずの頂点/共有グループ」に触れる危険がある

#### 評価

これは UI 層の「簡易実装」がドメイン整合性を壊している典型です。最優先で正規化すべきです。

#### 推奨対応

- `App.svelte` から直接 `Map.delete()` をやめる
- `DeleteFeatureUseCase` を UI から呼ぶ
- 可能なら削除も Command 化して Undo 対応する
- Dirty 更新とイベント通知をそこに寄せる

### 4.2 プロジェクト設定が永続化も反映もされていない [重大]

#### 現状コード 1: UI では値を持っている

`src/renderer/App.svelte`

```ts
let projectSettings = $state<WorldSettings>({ ...DEFAULT_SETTINGS });
let projectMetadata = $state<WorldMetadata>({ ...DEFAULT_METADATA });

function onSettingsSave(metaPatch: Partial<WorldMetadata>, settingsPatch: Partial<WorldSettings>): void {
  projectMetadata = { ...projectMetadata, ...metaPatch };
  projectSettings = { ...projectSettings, ...settingsPatch };
  projectMetadata = { ...projectMetadata, settings: projectSettings };
}
```

#### 現状コード 2: 保存時に metadata を捨てている

`src/renderer/application/SaveLoadUseCase.ts`

```ts
assembleWorld(): World {
  const metadata: WorldMetadata = {
    ...DEFAULT_METADATA,
    sliderMin: 0,
    sliderMax: 10000,
  };
```

#### 現状コード 3: 読み込み時も metadata を配っていない

`src/renderer/application/SaveLoadUseCase.ts`

```ts
private distributeWorld(world: World): void {
  this.addFeature.restore(world.features, world.vertices, world.sharedVertexGroups);
  this.manageLayers.restore(world.layers);
  // メタデータのスライダー範囲は将来のNavigateTimeUseCase拡張で対応
}
```

#### 現状コード 4: タイムラインも固定値

`src/renderer/presentation/components/TimelinePanel.svelte`

```ts
let sliderMin = $state(0);
let sliderMax = $state(10000);
```

#### 現状コード 5: 測量もデフォルト惑星固定

`src/renderer/App.svelte`

```ts
let surveyState = $state<SurveyModeState>(createSurveyState());
let surveyResult = $derived(computeSurveyResult(surveyState));
```

`src/renderer/infrastructure/rendering/surveyModeManager.ts`

```ts
export function createSurveyState(planet: PlanetParams = DEFAULT_PLANET): SurveyModeState {
  return { pointA: null, pointB: null, planet };
}
```

#### 問題

プロジェクト設定ダイアログは存在しますが、実質的にはローカル UI 状態です。

少なくとも以下が未接続です。

- プロジェクト名
- 説明
- タイムライン最小年・最大年
- 赤道長・扁平率
- グリッド設定
- 面スタイル系のデフォルト設定

#### 影響

- 保存しても設定が残らない
- 読み込んでも復元されない
- タイムライン範囲が変わらない
- 惑星パラメータを変えても測量結果が変わらない
- 「設定があるように見えるのに効いていない」という最悪の信頼性低下を招く

#### 評価

これは単なる未実装ではなく、UI が存在する分だけ危険です。ユーザーは「保存される」「反映される」と誤認します。

#### 推奨対応

- `SaveLoadUseCase` に `metadata/settings` の受け渡しを追加
- `World.metadata` をロード時に `App.svelte` 側へ反映
- `TimelinePanel` を `WorldMetadata.sliderMin/max` 連動に変更
- 測量の `PlanetParams` を `WorldSettings` から供給
- 設定変更時に Dirty 状態を立てる

### 4.3 自動バックアップが API 名不一致で壊れている [重大]

#### 現状コード 1: preload 側

`src/preload/index.ts`

```ts
contextBridge.exposeInMainWorld('api', api);
```

`src/preload/index.d.ts`

```ts
interface Window {
  api: FileAPI;
}
```

#### 現状コード 2: 呼び出し側

`src/renderer/App.svelte`

```ts
const content = await window.fileAPI.readFile(fromPath);
await window.fileAPI.writeFile(toPath, content);
await window.fileAPI.writeFile(getBackupFileName(filePath, 1), json);
```

#### 問題

公開名は `window.api`、使用側は `window.fileAPI` です。名前が一致していません。

しかも外側が `try/catch` なので、

- 例外は潰れる
- 画面上は何となく動く
- だがバックアップは作られない

という、発見しにくい壊れ方をしています。

#### 影響

- 「自動バックアップあり」と思っているのに、実際は無い
- データ消失時に最も困る系の不具合

#### 推奨対応

- `window.fileAPI` を `window.api` に統一
- Electron 実行環境での保存/バックアップ統合テストを追加
- バックアップ失敗時に UI 通知かログ可視化を入れる

### 4.4 Dirty 状態が一貫していない [高]

#### 現状コード

`src/renderer/App.svelte`

```ts
const unsubFeatureChanged = eventBus.on('feature:added', () => {
  markAsDirty();
});
```

```ts
function confirmUnsavedChanges(): boolean {
  if (!dirtyState.isDirty) return true;
  return confirm('未保存の変更があります。続行しますか？');
}
```

また、頂点削除は以下。

```ts
function onDeleteVertex(): void {
  // 頂点削除は後続で詳細化（コンテキストメニューと共通）
  ...
  vertexEdit.deleteVertexFromLine(...)
  ...
  refreshFeatureData();
}
```

#### 問題

Dirty 管理が「操作単位」ではなく「一部イベントと一部手動呼び出し」に分散しています。

現時点で少なくとも不安なのは以下です。

- 地物削除
- 頂点削除
- プロジェクト設定変更
- 一部の直接変更系処理

#### 影響

- 未保存変更警告が出たり出なかったりする
- バックアップ発火条件も `dirtyState.isDirty` に依存しているので、Dirty 漏れはバックアップ漏れにも直結する

#### 推奨対応

- Dirty 更新を UI ではなく UseCase / Command 成功時に集約する
- `feature:removed` など破壊的イベントにも必ず連動させる
- 可能なら「保存対象が変わる操作はすべて Command 経由」に寄せる

### 4.5 実ランタイムが DI 方針を使っていない [中]

#### 技術方針

`src/renderer/infrastructure/DIContainer.ts`

```ts
export class DIContainer {
  ...
}

export function getContainer(): DIContainer {
  if (!_instance) {
    _instance = new DIContainer();
  }
  return _instance;
}
```

#### 実際のランタイム

`src/renderer/presentation/state/appState.ts`

```ts
export const manageLayers = new ManageLayersUseCase();
export const navigateTime = new NavigateTimeUseCase();
export const addFeature = new AddFeatureUseCase();
...
export const saveLoad = new SaveLoadUseCase(repository, dialog, addFeature, manageLayers, navigateTime);
```

#### 問題

技術方針では DI 一元管理を掲げていますが、実際のアプリ起動経路は `appState.ts` のグローバルシングルトンです。

結果として、

- DIContainer は存在するが本番コードではほぼ使われない
- 初期化責務が二重化している
- 今後モック差し替えや設定差し込みをしたいときに、経路が二重で混乱する

#### 評価

今すぐ壊れる種類ではありません。ただし構造の一貫性は落ちています。

#### 推奨対応

- 実アプリの依存注入経路を `DIContainer` に寄せる
- 逆に `appState.ts` を残すなら、DIContainer を撤去して責務を一本化する

### 4.6 ファイル肥大化が進み、保守性が悪化している [中]

#### ガイドライン

`開発ガイド.md`

```md
- 1ファイルあたり300行を目安に分割（Svelteコンポーネントは200行以下、2.5節参照）
```

```md
- コンポーネントは200行以下を目安に分割。
```

`技術方針.md`

```md
- コンポーネント分割が自然で、各ファイルを200行以下に保ちやすい
```

#### 実測

- `src/renderer/App.svelte`: 1146 行
- `src/renderer/presentation/components/MapCanvas.svelte`: 556 行
- `src/renderer/infrastructure/persistence/JSONSerializer.ts`: 548 行

#### 問題

特に `App.svelte` は以下をまとめて持っています。

- ツール状態
- 選択状態
- Dirty 状態
- バックアップ
- 設定
- 分割
- 結合
- 測量
- 競合解決
- D&D 読込
- キーボードショートカット
- 各種 UI コールバック

これはルートコンポーネントの責務として過剰です。

#### 影響

- 1 つの変更の影響範囲が広がる
- 不具合修正時に無関係な回帰を生みやすい
- 状態の相互作用を追いにくい
- 「簡易実装」が混ざっても見逃されやすい

#### 推奨対応

- `App.svelte` を機能別コントローラへ分割
- 保存系、設定系、測量系、編集系、競合解決系を切り出す
- `MapCanvas.svelte` も入力制御と描画 UI を分離する

## 5. テスト面の評価

### 5.1 強い点

- 単体テストは非常に厚い
- 状態機械にもテストがある
- UI の主要シナリオは広く触れている

### 5.2 弱い点

現状の E2E はブラウザ実行ベースです。

`現状.md`

```md
Electron直接起動は ... 断念。
```

このため、以下は薄いです。

- preload API 公開名の整合
- IPC ハンドラとの実接続
- 実ファイル保存
- 自動バックアップ
- Electron 依存イベント

今回の `window.api` / `window.fileAPI` 不整合は、まさにこのギャップで取りこぼされた典型です。

## 6. 既知の残課題の読み方

`現状.md` には fixme ベースの既知課題もあります。

```md
5 fixme: 線地物追加、プロパティタブ自動切替、測量線残存、複数測量線、追加中Ctrl+Z頂点戻し
```

これらのうち、優先度の見方は以下が妥当です。

- 線地物追加
  - 機能コアに近いので中以上
- 追加中 Ctrl+Z
  - Undo の一貫性として中
- 測量線残存 / 複数測量線
  - 機能完成度としては低〜中
- プロパティタブ自動切替
  - UX 的には低

ただし、今回見つかった「削除経路」「設定永続化」「バックアップ破損」の方が優先順位は上です。

## 7. 優先順位付き改善提案

### 最優先

1. 地物削除を `DeleteFeatureUseCase` + Command 経由へ統一
2. Dirty 管理を UseCase / Command 成功時へ集約
3. 自動バックアップの API 名不一致修正

### 次点

4. `World.metadata/settings` の永続化・復元・UI 反映を一気通貫で接続
5. タイムラインと測量をプロジェクト設定連動に変更

### その次

6. `appState.ts` と `DIContainer.ts` の二重経路を一本化
7. `App.svelte` / `MapCanvas.svelte` の分割
8. Electron 直結の保存系統合テストを追加

## 8. 最終評価

このアプリは、企画・設計・ロジック・テストの地盤がかなり強いです。雑なプロトタイプではなく、きちんと作ろうとしているコードベースです。

ただし、現時点では「中心ロジックは良いが、アプリ統合の最後の 20% がまだ危ない」という状態です。特にデータ保全に関わる箇所が危ないため、安心して使えるアプリとして評価するにはまだ早いです。

短く言えば、以下です。

- 良い意味で: 中身は強い
- 厳しく言えば: 保存・設定・削除の統合が未完成

よって、現時点の評価は以下とします。

- 技術的ポテンシャル: 高い
- 現在の完成度: 中
- 保守性の将来見通し: このままだと悪化
- 優先すべき作業: 新機能追加より統合品質の是正
